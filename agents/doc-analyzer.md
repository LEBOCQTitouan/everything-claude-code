# Documentation Analyzer

Analyzes codebases to understand documentation needs. Produces structured analysis consumed by doc-generator, doc-validator, doc-reporter, diagram-generator.

## Reference Skills

- `skills/doc-analysis/SKILL.md` — methodology for public API, domain concepts, module boundaries
- Extraction: `symbol-extraction`, `behaviour-extraction`, `example-extraction`, `git-narrative`, `config-extraction`, `dependency-docs`, `failure-modes`

## Inputs

- `--scope=<path>` — directory to analyze (default: project root)
- Project root detected from `package.json`, `go.mod`, `pyproject.toml`, `Cargo.toml`, or `pom.xml`

## Pipeline

### Step 1: Project Profile

Detect language/framework, count source files per directory, identify entry points, classify size: **small** (<50 files) or **large** (50+), determine output format.

### Step 2: Module Inventory (symbol-extraction)

Per module: list source files, identify export boundaries (language-specific patterns), classify exports (function/class/type/interface/constant/enum), extract type signatures, count documented vs undocumented, list imports, resolve re-exports through barrels.

### Step 3: Domain Concept Extraction

Collect type/class/interface/enum names, split compound names, count frequency across files, keep terms in 3+ files, categorize as domain vs infrastructure, build glossary: term → files → inferred meaning.

### Step 4: Dependency Graph

Per module: list imports from other modules, annotate doc coverage (% documented), detect hub modules (fan-in >10), detect circular dependencies. Format as text graph with doc status.

### Step 5: Documentation Inventory

Per exported symbol: has doc comment? Has description/params/return/throws/example? Last modified date vs code date (git blame).

### Step 5b: Enrichment (priority modules only)

For high-priority modules (high fan-in, high change frequency, low doc coverage): behaviour extraction, config extraction, failure modes, git narrative, example extraction, dependency docs.

### Step 6: Diagram Opportunity Detection

Heuristics: 3+ cross-dep modules → flowchart, 3+ processing layers → data flow, 3+ types with inheritance → class diagram, cross-module flow → sequence, 3+ state enum → state machine, CI config → build pipeline.

Write `## Diagram Manifest` in `docs/ARCHITECTURE.md` (small) or `docs/DIAGRAM-MANIFEST.md` (large):

```markdown
| ID | Type | Scope | Title | Target Doc |
|----|------|-------|-------|------------|
```

## Output Structure

**Small (<50 files)**: `docs/ARCHITECTURE.md`, `docs/API-SURFACE.md`, `docs/DEPENDENCY-GRAPH.md`, `docs/domain/glossary.md`

**Large (50+)**: `docs/ARCHITECTURE.md` + `docs/api-surface/INDEX.md` + per-module, `docs/DEPENDENCY-GRAPH.md`, `docs/glossary/INDEX.md` + categories

Every file includes `<!-- Generated by doc-analyzer | Date: YYYY-MM-DD -->` header with `AUTO-GENERATED` markers. Cross-link with relative paths.

## Naming Expressiveness

Scan exports for generic names (Manager/Handler/Processor/Data/Info/Helper/Util). Per-module score: `(total - generic) / total`. Flag modules with score < 0.8.

## Commit Cadence

`docs: generate codebase analysis` after writing analysis files.

# Spec: Context-Aware Doc Generation at End of /implement (BL-056)

## Problem Statement

When `/implement` completes, the session holds rich context — spec decisions, grill-me rationale, design bounded contexts, TDD outcomes, and code review findings — that is lost when the session ends. The existing Phase 6 (Doc Updates) handles planned docs (CHANGELOG, ADRs, glossary) but does not capture architectural understanding: module purpose, design rationale, cross-module flows, and dependency changes. This context is most valuable immediately after implementation and prohibitively expensive to reconstruct later. Phase 7.5 captures it as persistent MODULE-SUMMARIES entries and Mermaid diagrams before the session closes.

## Research Summary

- **Diagrams should be generated on every code change, not on demand** — ensures a single source of truth, zero manual maintenance, and version-controlled architecture evolution (BSWEN CI/CD diagram guide)
- **Incremental doc updates outperform full regeneration** — the ReMe memory management pattern for agents uses merge-into-existing rather than rewrite, preventing loss of context-rich manual additions
- **Component diagrams > sequence diagrams for automation** — component/dependency diagrams can be derived from static analysis; sequence diagrams require runtime behavior understanding (harder to extract)
- **Mermaid renders natively in GitHub/GitLab** — diagrams committed as `.md` files with fenced code blocks are viewable directly in PRs and repo browsing
- **Context-rich docs have 75% higher onboarding value** — architecture docs that explain "why" (design rationale) alongside "what" (structure) significantly reduce onboarding time
- **Separate protected categories prevent regeneration conflicts** — when auto-generated and human-authored docs coexist, a clear ownership boundary prevents one from overwriting the other

## Decisions Made

| # | Decision | Rationale | ADR Needed? |
|---|----------|-----------|-------------|
| 1 | Create dedicated agent files (`agents/module-summary-updater.md`, `agents/diagram-updater.md`) instead of inline prompts | ECC conventions require frontmatter. implement.md is 364 lines — inline prompts risk 800-line limit. Separate files enable reuse. | No |
| 2 | Drop /doc-suite background trigger | `/doc-suite` is deprecated. Its doc-generator does full-file MODULE-SUMMARIES regeneration that would overwrite Phase 7.5's incremental updates. | No |
| 3 | Keep crate table, add per-module sections below AUTO-GENERATED markers | Preserves backward compatibility with doc-generator. Per-module entries live in a separate section. | No |
| 4 | Create "Feature Implementation Diagrams" category in INDEX.md, NOT registered in CUSTOM.md | CUSTOM.md diagrams are regenerated by /doc-suite, which would overwrite context-rich Phase 7.5 output. A separate category protects implementation diagrams. | Yes |
| 5 | Parent post-merge fixup pass for cross-linking | Subagents run in parallel and can't know each other's output. Parent reads diagram filenames after both complete and patches MODULE-SUMMARIES. | No |
| 6 | Add context checkpoint before Phase 7.5 consistent with Phases 4-7 | Phase 7.5 runs late in session when context exhaustion is most probable. Omitting contradicts the established pattern and creates a silent failure mode. | No |
| 7 | Both agents use model: haiku | Doc generation is lightweight. Haiku at 90% of Sonnet capability with 3x cost savings. | No |
| 8 | `## Supplemental Docs` always present (even if "none generated") | Consistent with `## E2E Tests` pattern. Hook checks section presence, not content. | No |
| 9 | Extend existing `doc-enforcement.sh` rather than new hook | Keeps all doc-related enforcement in one hook. Reduces hook proliferation. | No |
| 10 | Phase 7.5 subagent failures are non-blocking | Supplemental docs are valuable but not worth failing /implement. Commit successful output, record failure, proceed. | No |
| 11 | New agents have fewer than 4 workflow steps. TodoWrite not required per ECC convention. | Read context → update file → return. Under 4-step threshold. | No |
| 12 | diagram-updater output NOT registered in CUSTOM.md | Feature-specific diagrams are one-time snapshots derived from implementation context. Registered only in INDEX.md under "Feature Implementation Diagrams". doc-suite does not regenerate them. | No |

## User Stories

### US-001: Phase 7.5 Skeleton in /implement

**As a** developer running `/implement`, **I want** a Phase 7.5 inserted between Phase 6 (Doc Updates) and Phase 7 (implement-done.md), **so that** supplemental documentation is generated using full session context before that context is lost.

#### Acceptance Criteria

- AC-001.1: Given `/implement` Phase 6 completes, when the workflow proceeds, then Phase 7.5 begins before Phase 7
- AC-001.2: Given Phase 7.5, when it dispatches subagents, then two Task subagents (`module-summary-updater` and `diagram-updater`) are launched in parallel
- AC-001.3: Given both subagents complete, when their outputs are committed, then the parent runs a cross-link fixup pass before Phase 7 proceeds
- AC-001.4: Given Phase 7.5 runs, when the TodoWrite checklist and tasks.md entries exist, then a "Supplemental docs" entry appears between "Doc updates" and "Write implement-done.md"
- AC-001.5: Given Phase 7.5 is in the `/implement` command, when both subagents produce no changes (trivial feature), then Phase 7 proceeds with "No supplemental docs generated" recorded
- AC-001.6: Given Phase 7.5 is about to start, when the parent runs the context checkpoint, then it follows the same graceful-exit protocol as Phases 4-7 (>= 85% save-and-exit, >= 75% warning)
- AC-001.7: Given implement.md Phase 0 re-entry logic, when it lists Post-TDD phases for resume, then "Supplemental docs" appears between "docs" and "implement-done"
- AC-001.8: Given one subagent fails and the other succeeds, when the parent handles the result, then the successful output is committed, the failure is recorded in implement-done.md Supplemental Docs section, and Phase 7 proceeds
- AC-001.9: Given Phase 7.5 dispatches Task subagents, when each Task is created, then allowedTools is specified matching the agent's frontmatter tools list
- AC-001.10: Given both subagents are dispatched, when the parent proceeds to the cross-link fixup, then it has confirmed both tasks reached a terminal status (completed or failed) first
- AC-001.11: Given implement.md is modified for Phase 7.5, then these sections are also updated: Phase 0 re-entry list, Phase 2 TodoWrite template, Phase 8 stop-gate verification list, Related Agents section
- AC-001.12: Given implement-done.md is written in Phase 7, then the `## Supplemental Docs` section uses this exact table format: `| Subagent | Status | Output File | Commit SHA | Notes |` with one row per subagent. Status is "success" or "failed". Commit SHA is the fixup commit hash or "N/A" on failure. Notes contains failure reason when status is "failed", or "--" on success.

#### Dependencies

- Depends on: none

### US-002: Module Summary Updater Agent

**As a** developer running `/implement`, **I want** MODULE-SUMMARIES.md updated with entries for every module changed during the TDD loop, **so that** design rationale and cross-links are captured while session context is fresh.

#### Acceptance Criteria

- AC-002.1: Given `agents/module-summary-updater.md` exists, when it has YAML frontmatter, then it contains name, description, tools (Read, Write, Edit, Grep, Glob), and model (haiku)
- AC-002.2: Given the TDD loop changed files in crates, when Subagent A runs, then `docs/MODULE-SUMMARIES.md` has updated or new entries for each changed module below the AUTO-GENERATED markers
- AC-002.3: Given an existing entry in MODULE-SUMMARIES.md, when the module was modified, then only that entry is updated (no full-file regeneration)
- AC-002.4: Given a new module was created, when Subagent A runs, then a new entry is appended with: purpose (1-2 sentences), key functions/types, spec cross-link, ADR cross-link (if applicable), design rationale note
- AC-002.5: Given no files changed in the TDD loop, when Subagent A runs, then it records "No module changes" and exits cleanly
- AC-002.6: Given MODULE-SUMMARIES.md does not exist, when Subagent A runs, then it creates the file with the standard crate table header and adds per-module entries below
- AC-002.7: Given Subagent A finishes with changes, when it returns to the parent, then the parent commits: `docs: update MODULE-SUMMARIES for <feature>`
- AC-002.8: Given MODULE-SUMMARIES.md has AUTO-GENERATED markers, when entries are added or updated, then they are placed AFTER `<!-- END AUTO-GENERATED -->` and wrapped in `<!-- IMPLEMENT-GENERATED -->` / `<!-- END IMPLEMENT-GENERATED -->` markers
- AC-002.9: A "module" means a Rust crate from the workspace. One MODULE-SUMMARIES entry per crate that had files modified during the TDD loop.
- AC-002.10: Given a design rationale note in a module entry, then it references at least one spec decision number (D-N) or grill-me question ID

#### Dependencies

- Depends on: US-001

### US-003: Diagram Updater Agent

**As a** developer running `/implement`, **I want** Mermaid diagrams generated or updated for the implemented feature, **so that** architectural flows and dependencies are visually documented while design context is fresh.

#### Acceptance Criteria

- AC-003.1: Given `agents/diagram-updater.md` exists, when it has YAML frontmatter, then it contains name, description, tools (Read, Write, Edit, Grep, Glob), and model (haiku)
- AC-003.2: Given new cross-module flows were introduced, when Subagent B runs, then a sequence diagram is created in `docs/diagrams/`
- AC-003.3: Given a new decision path or state machine was added, when Subagent B runs, then a flowchart is created
- AC-003.4: Given new crate-level dependencies were introduced, when Subagent B runs, then a separate feature-specific dependency diagram is created (NOT modifying `module-dependency-graph.md`)
- AC-003.5: Given a new bounded context or port/adapter pair was added, when Subagent B runs, then a C4 component diagram is created
- AC-003.6: Given any diagram was created or updated, when Subagent B finishes, then `docs/diagrams/INDEX.md` is updated with the entry under "Feature Implementation Diagrams" category
- AC-003.7: Given no diagram is warranted (trivial change), when Subagent B runs, then it skips and records "No diagram generated"
- AC-003.8: Given each diagram file, when written, then it uses the existing format: generator comment header + Mermaid fenced code block
- AC-003.9: Given Subagent B finishes with changes, when it returns to the parent, then the parent commits: `docs(diagrams): add <feature> diagrams`
- AC-003.10: Given diagram-updater creates a diagram file, when it writes the generator comment header, then the header uses `diagram-updater` as the agent name (e.g., `<!-- Generated by diagram-updater | Date: ... | Source: ... -->`)
- AC-003.11: Given `docs/diagrams/module-dependency-graph.md` is owned by diagram-generator, then diagram-updater MUST NOT modify this file. New crate dependencies get a separate file.
- AC-003.12: Diagram trigger heuristics: cross-module flow = files changed span 2+ crates AND involve a new public function call chain; state machine = enum with 3+ variants added or modified; bounded context = new crate directory created under `crates/`
- AC-003.13: Given diagram-updater creates a diagram file, then it MUST NOT register the diagram in `docs/diagrams/CUSTOM.md`. Feature implementation diagrams are one-time snapshots and are not subject to doc-suite regeneration.
- AC-003.14: Given diagram-updater registers a new diagram in INDEX.md, then the entry is placed under a `## Feature Implementation Diagrams` section (created after `## Command Workflow Diagrams` if absent). The Coverage table is updated to include the new category.

#### Dependencies

- Depends on: US-001

### US-004: Cross-Link Fixup Pass

**As a** developer reviewing documentation, **I want** MODULE-SUMMARIES entries to link to related diagrams and vice versa, **so that** I can navigate between textual and visual documentation.

#### Acceptance Criteria

- AC-004.1: Given both subagents complete, when the parent runs the fixup pass, then MODULE-SUMMARIES entries for changed modules include links to any diagrams created by Subagent B
- AC-004.2: Given Subagent B produced no diagrams, when the fixup pass runs, then no cross-links are added (no-op)
- AC-004.3: Given the fixup pass modifies MODULE-SUMMARIES, when it completes, then the change is included in a separate fixup commit: `docs: cross-link MODULE-SUMMARIES to diagrams for <feature>`

#### Dependencies

- Depends on: US-002, US-003

### US-005: Extend implement-done.md Schema

**As a** developer reviewing implementation output, **I want** implement-done.md to include a `## Supplemental Docs` section, **so that** the Phase Summary captures what Phase 7.5 produced.

#### Acceptance Criteria

- AC-005.1: Given Phase 7.5 completed, when Phase 7 writes implement-done.md, then a `## Supplemental Docs` section is present with a table listing: doc file, type (module-summary/diagram), what changed
- AC-005.2: Given Phase 7.5 produced no changes, when Phase 7 writes implement-done.md, then `## Supplemental Docs` contains "No supplemental docs generated — change scope did not warrant module summary or diagram updates"
- AC-005.3: Given the `## Supplemental Docs` section schema, when it is in the implement-done.md template in commands/implement.md, then it appears between `## ADRs Created` and `## Subagent Execution`

### US-006: Extend Stop-Gate Hook for Supplemental Docs

**As a** workflow enforcer, **I want** the doc-enforcement hook to verify `## Supplemental Docs` is present in implement-done.md, **so that** Phase 7.5 output is guaranteed to be recorded.

#### Acceptance Criteria

- AC-006.1: Given phase is `done`, when `doc-enforcement.sh` runs, then it checks for `## Supplemental Docs` heading in implement-done.md
- AC-006.2: Given `## Supplemental Docs` is missing, when the hook runs, then it emits a WARNING (not an error, consistent with existing behavior)
- AC-006.3: Given `## Supplemental Docs` is present, when the hook runs, then it passes silently

## Constraints

- `commands/implement.md` must stay under 800 lines after changes
- Both new agents must have valid YAML frontmatter per ECC conventions (name, description, tools, model)
- MODULE-SUMMARIES.md AUTO-GENERATED markers must be preserved — per-module entries go AFTER `<!-- END AUTO-GENERATED -->` wrapped in IMPLEMENT-GENERATED markers
- Feature implementation diagrams must NOT be registered in CUSTOM.md (protected from regeneration)
- diagram-updater must NOT modify `module-dependency-graph.md` (owned by diagram-generator)
- Hook changes must follow existing patterns: `ECC_WORKFLOW_BYPASS`, `set -uo pipefail`, warning-only
- Agent model must be `haiku` for both new agents
- diagram-updater header must use `diagram-updater` as agent name (not `diagram-generator`)

## Non-Requirements

- Modifying the /doc-suite pipeline or its agents
- Creating a /doc-suite replacement
- Triggering /doc-suite as background process (dropped — deprecated, overwrites Phase 7.5 output)
- Regenerating full MODULE-SUMMARIES.md file
- Re-reading source code in subagents (derive from session context)
- Any Rust code changes
- Modifying `module-dependency-graph.md` (owned by diagram-generator)

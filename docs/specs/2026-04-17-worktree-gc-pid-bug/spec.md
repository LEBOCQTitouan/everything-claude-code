# Spec: Fix Worktree GC PID Bug (BL-150)

## Problem Statement

`ecc-workflow worktree-name` generates worktree names using `process::id()` — the PID of the ecc-workflow binary subprocess, which exits immediately after printing the name. The worktree GC subsystem (`is_worktree_stale()` in `ecc-app/src/worktree/mod.rs:64`) checks `kill -0 <pid>` to determine liveness. Since the PID is always dead, every ECC worktree is considered stale from the moment it's created. When `session_start` fires GC (via `best_effort_gc` in `lifecycle.rs:21`) from any parallel session in the same repo, the worktree is deleted. A secondary defect — `.unwrap_or(0)` on `unmerged_commit_count` in `gc.rs:64-65` — silently defaults to "0 unmerged" when the query fails, bypassing the unmerged-commit safety net. Impact: complete loss of in-progress spec/design/implementation artifacts with no recovery path. Observed twice in a single 4-hour session.

## Research Summary

- Web research skipped: internal ECC bug with no external pattern.
- `std::os::unix::process::parent_id()` returns the PPID as `u32` — available on Unix since Rust 1.64. `ecc-workflow` is already Unix-only.
- `unwrap_or(0)` on error-prone queries is a known Rust anti-pattern; `unwrap_or(u64::MAX)` is the standard fail-safe default.
- File modification time via `stat` can check `.git` file recency through `ShellExecutor` for hexagonal purity.

## Decisions Made

| # | Decision | Rationale | ADR Needed? |
|---|---|---|---|
| 1 | Use `parent_id()` for PID in worktree names | Captures Claude Code session PID; strictly safer for standalone use too | No |
| 2 | Change `.unwrap_or(0)` to `.unwrap_or(u64::MAX)` | Fail-safe: assume unmerged when count query fails | No |
| 3 | Add 30-minute recency check via `.git` metadata | Defense-in-depth: recently-modified worktrees are never stale | No |
| 4 | Route recency check through ShellExecutor | Hexagonal purity: ecc-app must not call std::fs directly | No |
| 5 | Rename `WorktreeError` in ecc-app → `WorktreeGcError` | Closes CONV-006 duplicate name finding | No |
| 6 | Add targeted tests for all 3 fixes | Current tests don't cover the failure path | No |
| 7 | Add temporary CLAUDE.md gotcha about parallel sessions | User protection until fix ships | No |

## User Stories

### US-001: Fix PID source in worktree name generation

**As a** contributor, **I want** worktree names to embed the Claude Code session PID (not the ecc-workflow subprocess PID), **so that** GC correctly identifies active worktrees as non-stale.

#### Acceptance Criteria
- AC-001.1: Given `crates/ecc-workflow/src/commands/worktree_name.rs`, when generating a worktree name, then `std::os::unix::process::parent_id()` is used instead of `process::id()`.
- AC-001.2: Given a worktree name generated by `ecc-workflow worktree-name`, then the embedded PID equals `std::os::unix::process::parent_id()` (unit-testable without a live Claude Code session).
- AC-001.3: Given existing tests for `WorktreeName::generate()`, then they still pass.

#### Dependencies
- Depends on: none

### US-002: Fail-safe unmerged commit count default

**As a** contributor, **I want** the GC to skip worktrees when the unmerged-commit-count query fails, **so that** I/O failures don't silently delete worktrees with unmerged work.

#### Acceptance Criteria
- AC-002.1: Given `crates/ecc-app/src/worktree/gc.rs:64-65`, then `.unwrap_or(0)` is replaced with `.unwrap_or(u64::MAX)`.
- AC-002.2: Given a mock `WorktreeManager` that returns `Err` for `unmerged_commit_count`, when GC runs with `force=false`, then the worktree is skipped (not removed).
- AC-002.3: Given a mock `WorktreeManager` that returns `Ok(0)`, when GC runs, then the stale worktree is removed (unchanged behavior).

#### Dependencies
- Depends on: none

### US-003: Add liveness check for recently-modified worktrees

**As a** contributor, **I want** GC to skip worktrees whose `.git` file was modified within the last 30 minutes, **so that** actively-used worktrees are protected even if PID checks are unreliable.

#### Acceptance Criteria
- AC-003.1: Given `is_worktree_stale()`, then a new check queries `.git` file modification time via `ShellExecutor`.
- AC-003.2: Given a worktree modified <30 minutes ago, when GC runs, then the worktree is NOT stale.
- AC-003.3: Given a worktree modified >30 minutes ago AND PID dead AND age >24h, then the worktree IS stale.
- AC-003.4: Given the stat command fails, then the recency guard is not applied — staleness is determined by age and PID checks only (existing behavior preserved on stat failure).
- AC-003.5: Given the `is_worktree_stale` function, then it accepts the worktree path as an additional parameter.
- AC-003.6: The complete staleness formula is: `stale = (age > 24h OR pid_dead) AND modified > 30min`. A worktree is stale only when BOTH the existing staleness condition (age OR PID) AND the recency condition (not recently modified) are true. This is the definitive truth table for all combinations.

#### Dependencies
- Depends on: none

### US-004: Rename duplicate WorktreeError (CONV-006)

**As a** maintainer, **I want** the `WorktreeError` in `ecc-app` renamed to `WorktreeGcError`, **so that** error types are unambiguous.

#### Acceptance Criteria

- AC-004.2: All call sites in `ecc-app` updated.
- AC-004.3: `cargo build --workspace` succeeds.

#### Dependencies
- Depends on: none

### US-005: Add temporary CLAUDE.md gotcha

#### Acceptance Criteria
- AC-005.1: CLAUDE.md gotchas section contains: "TEMPORARY (BL-150): Do not run parallel Claude Code sessions in the same repo — worktree GC may delete active worktrees. Remove after BL-150 ships."

#### Dependencies
- Depends on: none

## Constraints

- `parent_id()` is Unix-only. `ecc-workflow` is already Unix-only.
- `is_worktree_stale` signature change requires updating call sites (gc.rs:57 + tests).
- Recency check routes through `ShellExecutor`, not raw `std::fs`.
- Existing 14 tests must pass unchanged. New tests are additive.

## Non-Requirements

- Renaming `WorktreeError` in `ecc-ports`.
- Env var injection (CLAUDE_SESSION_PID).
- Windows support.
- Recovering lost spec/design artifacts from prior sessions.

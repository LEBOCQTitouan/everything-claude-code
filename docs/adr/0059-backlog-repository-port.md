# ADR 0059: Promote Backlog to Full Hexagonal Concern with Port Traits

## Status

Accepted

## Context

The backlog subsystem was the last major domain to use the generic `FileSystem` port for persistence. This caused:

- Duplicated logic: `ecc-workflow` reimplemented `compute_next_id` and `update_backlog_index` with raw `std::fs`
- Untestable filtering: the `/spec` command implemented worktree+lock filtering as ad-hoc Claude tool calls in Markdown
- Manual index: `BACKLOG.md` required manual status updates (98 commits in 180 days with 0.00 co-change ratio to Rust code)
- ISP violation: a single 10-method `BacklogRepository` trait would force all consumers to depend on lock, entry, and index operations

## Decision

Split backlog persistence into 3 ISP-compliant port traits:

- `BacklogEntryStore` — CRUD for BL-*.md entry files (4 methods)
- `BacklogLockStore` — session lock lifecycle (4 methods)
- `BacklogIndexStore` — BACKLOG.md index I/O (2 methods)

A single `FsBacklogRepository` struct in `ecc-infra` implements all three via `&dyn FileSystem`. Each use case accepts only the trait(s) it needs.

Status reconciliation uses the Kubernetes-style pattern: derive `in-progress` from worktree existence rather than dual-writing status.

## Consequences

- `ecc-app::backlog` functions accept port traits instead of `&dyn FileSystem`
- `ecc-workflow::commands::backlog` delegates to `ecc-app` instead of reimplementing
- `BACKLOG.md` status is generated by `ecc backlog reindex` from worktree state
- New `ecc backlog list --available` CLI command replaces Markdown-based filtering
- `InMemoryBacklogRepository` in `ecc-test-support` enables deterministic testing
- `LockFile` domain value object codifies the lock file format (worktree name + ISO 8601 timestamp)

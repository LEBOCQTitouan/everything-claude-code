# 0015. Feature Implementation Diagrams Category

Date: 2026-03-23

## Status

Accepted

## Context

`/implement` Phase 7.5 generates Mermaid diagrams that capture cross-module flows, state machines, and dependency changes while implementation context is fresh. These diagrams are one-time snapshots tied to a specific feature implementation — they preserve design rationale that would be expensive to reconstruct later.

The existing diagram ecosystem has two categories: Auto-Generated (derived from codebase analysis, regenerated by `/doc-suite`) and Custom-Registered (listed in `CUSTOM.md`, also regenerated by `/doc-suite`). Placing Phase 7.5 diagrams in either category would cause `/doc-suite` to overwrite context-rich implementation snapshots with generic regenerated output.

## Decision

Create a third diagram category — "Feature Implementation Diagrams" — in `docs/diagrams/INDEX.md`. Diagrams in this category:

- Are generated by `/implement` Phase 7.5's `diagram-updater` agent
- Are registered in INDEX.md under the `## Feature Implementation Diagrams` section
- Are NOT registered in `CUSTOM.md`
- Are NOT subject to `/doc-suite` regeneration

## Consequences

**Positive:**

- Phase 7.5 diagrams are preserved across `/doc-suite` runs
- Clear ownership boundary: `diagram-generator` owns Auto-Generated and Custom-Registered; `diagram-updater` owns Feature Implementation
- INDEX.md remains the single source of truth for all diagram categories

**Negative:**

- Feature Implementation Diagrams may become stale as the codebase evolves (they are snapshots, not living documents)
- A third category adds minor cognitive overhead when navigating the diagram index

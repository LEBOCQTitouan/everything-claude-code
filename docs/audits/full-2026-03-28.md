## Executive Summary

The ECC codebase demonstrates excellent architectural discipline: the hexagonal crate structure has zero dependency direction violations, the domain crate maintains perfect I/O purity, and port-based testing enables a 94.4% pure unit test ratio across 1,404 tests. The Stable Dependencies Principle is correctly followed, and the codebase has no circular dependencies.

The primary risks cluster around three themes: (1) **file growth** — 5 files exceed the 800-line limit, with `integration.rs` at 3.4x, driven by inline tests inflating production files; (2) **observability gaps** — all 48 `warn!` calls are silent by default and 38 error-discard sites emit no diagnostics, making production failures invisible; (3) **documentation staleness** — two files are entirely from the TypeScript era with zero valid references, and ARCHITECTURE.md counts are 47-63% off. The `ecc-workflow` binary bypasses the port system entirely, creating a parallel I/O pathway that cannot use the project's test doubles.

Cross-domain correlation identified 7 compound risks, including 3 CRITICAL escalations where hotspot files combine with untested code, swallowed errors, or boundary violations.

## Cross-Domain Correlations

### [CORR-001] Hotspot + Zero Unit Tests: ecc-workflow commands
- **Severity**: CRITICAL
- **Sources**: [EVL-002] + [TEST-002] + [TEST-003]
- **Correlation**: `ecc-workflow/src/main.rs` has the highest hotspot score (0.909) and highest complexity density (0.215). The 12 command files beneath it (1,263 lines total) have zero unit tests. `memory_write.rs` alone is 449 lines with no unit tests and 26 direct `std::fs` calls.
- **Evidence**: Hotspot score 0.909 (rank 1), 20 changes in 180 days, 0 `#[test]` in any command file. All coverage comes from binary-spawning integration tests in the 2,750-line monolith.
- **Risk**: The most frequently changed code with the highest complexity has the slowest test feedback loop and no fast-fail safety net. Logic errors require full build + process invocation to detect.
- **Remediation**: Extract pure logic functions from command handlers (format builders, decision logic, argument parsers) and test them inline. The I/O shell stays thin and untested.

### [CORR-002] Swallowed Errors + Silent Logging: production failures invisible
- **Severity**: CRITICAL
- **Sources**: [ERR-001] + [ERR-002] + [OBS-001] + [OBS-002]
- **Correlation**: 38 `Err(_)` arms silently discard errors (OBS-002). The 48 `warn!` calls that could surface them are invisible because `env_logger` defaults to `ERROR` level (OBS-001). The REPL `run_repl` result is discarded with `let _ =` (ERR-001). stdin read failures produce empty strings (ERR-002).
- **Evidence**: `env_logger::Builder::from_default_env()` with no `default_filter_or`. Zero `debug!` or `info!` calls. `--verbose` is a no-op.
- **Risk**: A chain of silent failures can propagate: corrupted state file -> empty stdin -> phase gate passes incorrectly -> workflow enters wrong phase. No diagnostic breadcrumb exists at any point in this chain.
- **Remediation**: Set default log level to `WARN` (one-line fix). Add `log::warn!` to the 38 error-discard sites. Add `log::debug!` at operation boundaries for `--verbose`.

### [CORR-003] Hotspot + Boundary Violation: ecc-workflow bypasses hexagonal architecture
- **Severity**: CRITICAL
- **Sources**: [EVL-002] + [ARCH-001] + [OBS-004]
- **Correlation**: `ecc-workflow` is the top hotspot crate (46 changes, top 3 hotspot scores). It bypasses the port system with 60+ direct `std::fs` calls. It has no logging infrastructure. Combined: the most active, most complex code path operates outside the architecture's quality guardrails.
- **Evidence**: No `ecc-ports` dependency. No `log` dependency. No `--verbose` flag. 60+ `std::fs::` calls across 14 files.
- **Risk**: Regressions in workflow state management (the gate system that enforces the spec pipeline) are hard to test, hard to debug, and hard to observe in production. This is the codebase's highest-risk blind spot.
- **Remediation**: Phase 1: Add logging to ecc-workflow. Phase 2: Extract pure logic from commands for unit testing. Phase 3 (long-term): Consider introducing port traits for the workflow binary.

### [CORR-004] Bus Factor 1 + Hotspot concentration
- **Severity**: HIGH
- **Sources**: [EVL-010] + [EVL-011] + [DOC-001] + [DOC-002]
- **Correlation**: Bus factor is 1 across all Rust code. The two stale TypeScript-era docs (DEPENDENCY-GRAPH.md, glossary.md) would mislead any new contributor trying to understand the codebase. Combined: if the sole contributor becomes unavailable, onboarding documentation actively hinders knowledge transfer.
- **Evidence**: 1 contributor for all Rust files. 174 lines of dead TypeScript references in DEPENDENCY-GRAPH.md. 27 dead `.ts` file links in glossary.md.
- **Risk**: Knowledge transfer failure. A new contributor would read the stale docs and build an incorrect mental model.
- **Remediation**: Regenerate DEPENDENCY-GRAPH.md for Rust crates. Update glossary.md file references. These are the highest-value documentation fixes for bus factor mitigation.

### [CORR-005] Convention drift + High churn: error type inconsistency in ecc-app
- **Severity**: MEDIUM
- **Sources**: [CONV-001] + [CONV-004] + [ERR-006] + [ERR-007]
- **Correlation**: `ecc-app` has 92 changes in 180 days (highest churn). It uses 3 competing error strategies (`Result<T, String>`, `anyhow::Result`, typed enums) and 3 competing frontmatter parsing implementations. High churn amplifies inconsistency as each new feature picks whichever pattern the author encounters first.
- **Evidence**: 14 functions return `Result<T, String>`, 12 use `anyhow::Result`, 54 use typed enums. 4 different frontmatter parsing implementations.
- **Risk**: New features in ecc-app will continue to diverge unless conventions are enforced. The inconsistency makes refactoring harder and error handling unreliable.
- **Remediation**: Define a canonical error type per ecc-app module. Consolidate frontmatter parsing to `serde_yaml`. Remove `anyhow` from ecc-app's public API.

### [CORR-006] Low coverage + High fan-in: FileSystem trait ISP violation
- **Severity**: MEDIUM
- **Sources**: [ARCH-002] + [TEST-007] + [EVL-005]
- **Correlation**: The `FileSystem` trait has 20 methods (ISP violation). `ecc-infra` adapters implementing it have zero tests. The trait is depended upon by 4 crates (high fan-in). Any change to the trait signature propagates to all adapters and test doubles.
- **Evidence**: 20 methods on FileSystem. 0 tests on OsFileSystem. 4 crates depend on ecc-ports.
- **Risk**: A method signature change on FileSystem requires coordinated updates across 4 crates with no adapter-level regression tests to catch mistakes.
- **Remediation**: Split FileSystem into sub-traits. Add smoke tests for OsFileSystem.

### [CORR-007] Security injection risk + No test coverage: notification hook
- **Severity**: HIGH
- **Sources**: [SEC-001] + [TEST-004]
- **Correlation**: The osascript/PowerShell injection in `tier2_notify.rs` (SEC-001) interpolates user-controllable strings into shell commands. The `ecc-cli` has zero tests (TEST-004), and the notification hook handler has no dedicated unit tests for the command construction path.
- **Evidence**: `format!("osascript -e 'display notification \"{}\"'", message)` with no escaping. No tests for the notification handler.
- **Risk**: A tool name or session ID containing `'` or `"` characters can break out of the osascript command string. No test catches this.
- **Remediation**: Escape interpolated strings in the osascript/PowerShell command. Add a unit test with adversarial input (e.g., `tool'; rm -rf /; echo '`).

## Prioritized Remediation Roadmap

### Immediate (CRITICAL)

1. **[CORR-002] Fix default log level** — Change `env_logger::Builder::from_default_env()` to `from_env(Env::default().default_filter_or("warn"))`. One-line fix that makes all 48 existing warn! calls visible. Addresses OBS-001 + ERR compound risk.

2. **[CORR-001] Extract pure logic from ecc-workflow commands** — Start with `memory_write.rs` (449 lines): extract format builders as pure functions, add unit tests. Unblocks fast test feedback for the highest-risk code.

3. **[DOC-001/DOC-002] Regenerate stale TypeScript-era docs** — Delete and rewrite `DEPENDENCY-GRAPH.md` for Rust crates. Update all `.ts` references in `glossary.md`. Critical for bus factor mitigation.

### Short-term (HIGH)

4. **[SEC-001] Fix osascript injection** — Escape interpolated strings in `tier2_notify.rs`. Add adversarial input test.

5. **[SEC-002] Bound stdin read in ecc-workflow** — Add 1MB limit matching ecc-cli's existing pattern.

6. **[CONV-001/ERR-006] Standardize error types in ecc-app** — Replace `Result<T, String>` with typed error enums. Remove `anyhow` from ecc-app public API.

7. **[CONV-004] Consolidate frontmatter parsing** — Standardize on `serde_yaml` for all frontmatter.

8. **[DOC-003-011] Regenerate ARCHITECTURE.md** — Update agent (49), command (23), skill (109) counts. Add missing crates to MODULE-SUMMARIES, bounded-contexts, getting-started, and module-dependency-graph.

9. **[TEST-001] Split integration.rs** — Decompose 2,750-line monolith into per-command test files.

10. **[OBS-004] Add logging to ecc-workflow** — Add `log`/`env_logger` dependency and `debug!` calls.

### Medium-term (MEDIUM)

11. **Split oversized files** — Decompose `validate.rs`, `dev.rs`, `merge/helpers.rs`, `install/global.rs` into submodules.

12. **Add unit tests to coverage gaps** — tier3_session tracking/lifecycle, merge/mod.rs orchestrator, ecc-infra adapters.

13. **Enrich WorkflowState aggregate** — Move mutation methods into domain, replace String fields with typed enums.

14. **Add debug! calls** — Make `--verbose` produce meaningful output.

15. **Add failure banners** — Explicit "Install failed" / "Dev switch failed" output before exit(1).

## Top 5 Recommendations

1. **Fix the logging default** (CORR-002) — A single line change (`default_filter_or("warn")`) immediately surfaces 48 existing warning calls and makes production failures visible. Highest signal-to-effort ratio of any finding.

2. **Add unit tests to ecc-workflow commands** (CORR-001) — Extract pure logic from the 12 untested command files. The workflow gate system is the spine of the spec pipeline; it needs fast, reliable test coverage.

3. **Regenerate stale documentation** (DOC-001/002, DOC-003-011) — Two files are entirely from the TypeScript era. ARCHITECTURE.md counts are 47-63% wrong. Run `/doc-suite` to regenerate.

4. **Fix security injection in notifications** (SEC-001) — Escape user-controllable strings in osascript/PowerShell commands. Add adversarial test.

5. **Standardize error handling in ecc-app** (CONV-001/ERR-006) — Replace `Result<T, String>` and `anyhow` in library API with typed error enums. This unlocks proper error translation at the CLI boundary and improves user-facing messages.

# Codebase Health Audit — 2026-03-14

**Project**: Everything Claude Code (ECC) — Rust CLI
**Scope**: `/crates/` (109 `.rs` files, ~22,400 lines, 6 crates)
**Architecture**: Hexagonal (domain → ports → app → infra → cli)
**Prior audits**: None (first audit)
**Overall health grade**: **B+**

---

## Executive Summary

The ECC Rust codebase has exemplary architectural discipline — clean hexagonal layering, zero circular dependencies, pure domain logic, and well-designed port abstractions. Architecture scores an **A**. However, operational robustness is undermined by a systemic silent-error-swallowing pattern (14 `let _ =` sites on write paths), the complete absence of a logging framework, and four untested orchestration modules totaling 1,274 lines. These gaps are amplified by a bus factor of 1 across the entire codebase.

## Phase 1: Evolutionary Analysis

### [EVOLUTION-001] Bus Factor = 1 Across Entire Rust Codebase — CRITICAL

- **Location**: All 109 files in `crates/`
- **Evidence**: Every file has exactly 1 unique author across all 26 commits touching Rust source. No second contributor has committed to any `.rs` file.
- **Risk**: Total knowledge concentration. If the sole contributor is unavailable, 22,400 lines of Rust have zero institutional knowledge backup.
- **Remediation**: Prioritize onboarding a second contributor. Start with pair programming on hook handlers and session management (highest complexity modules).

### [EVOLUTION-002] 14 Files Exceed 500 Lines — HIGH

- **Location**: `ecc-app/src/session/aliases.rs` (777 lines, 97% of 800-line limit), plus 13 others
- **Evidence**: 13 of 14 oversized files are in `ecc-app`. The largest is approaching the 800-line hard limit.
- **Risk**: Next feature addition to `session/aliases.rs` will breach the guideline.
- **Remediation**: Proactively split `session/aliases.rs`, `session/mod.rs` (697), and `config/audit/checks.rs` (695).

### [EVOLUTION-003] ecc-app Crate Dominates Codebase — MEDIUM

- **Location**: `ecc-app/` (44 files, 13,530 lines — 60% of all Rust code)
- **Evidence**: Contains 13 of 14 oversized files. All 6 prior splitting refactors targeted `ecc-app`.
- **Remediation**: Consider extracting `claw`, `hook`, and `session` into their own crates.

### [EVOLUTION-004] Hook Handler Modules Have Highest Complexity Density — MEDIUM

- **Location**: `tier2_tools.rs` (0.104), `tier1_simple/handlers.rs` (0.144), `tier3_session/helpers.rs` (0.091)
- **Evidence**: Average complexity density 0.113 — roughly 2x the codebase average.
- **Remediation**: Extract conditional logic into smaller helpers or use table-driven approaches.

### [EVOLUTION-005] Cross-Crate Coupling Between App and Domain — LOW

- **Location**: `merge` and `session` modules show consistent app-domain co-change coupling
- **Evidence**: 2 co-changes per pair. Expected in hexagonal architecture but density suggests tight binding.

### [EVOLUTION-006] Successful Refactoring Pattern — LOW (Positive)

- **Evidence**: 7 deliberate splitting refactors kept all files under 800 lines despite initial sizes exceeding 2,400 lines.

- **Location**: `ecc-cli/src/commands/install.rs:108-131`
- **Evidence**: Uses `std::path::PathBuf::exists()` and `std::env::current_exe()` directly instead of `FileSystem` and `Environment` port traits.
- **Remediation**: Move into `ecc-app` as a use case accepting `&dyn FileSystem` and `&dyn Environment`.

#### [ARCH-002] Duplicated `ECC_PACKAGE_IDENTIFIERS` Constant — MEDIUM

- **Location**: `ecc-domain/src/config/audit.rs:73` and `ecc-domain/src/config/merge.rs:30`

#### [ARCH-003] FileSystem Trait Has 13 Methods — MEDIUM

#### [ARCH-004] CLI Imports `ClawModel` Directly From Domain — LOW

- **Location**: `ecc-cli/src/commands/claw.rs:5`
- **Remediation**: Re-export through `ecc-app`.

#### [ARCH-005] Mirror Module Naming Convention — LOW

- **Evidence**: `config`, `detection`, `session`, `claw` modules exist in both domain and app. Intentional but requires familiarity.

#### [ARCH-006] CLI Uses Domain Time Directly — LOW

- **Location**: `ecc-cli/src/commands/install.rs:69-80`
- **Evidence**: `ecc_domain::time::datetime_from_epoch` used directly in CLI layer.

### Security (Grade: A-)

#### [SEC-001] User-Controlled File Paths Passed to Shell Commands — MEDIUM

- **Location**: `ecc-app/src/hook/handlers/tier2_tools.rs:46, 79, 84, 130, 196-227`
- **Evidence**: `file_path` extracted from JSON stdin passed as arguments to biome, prettier, tsc, gofmt, ruff. Uses `Command::new()` with separate args (not shell strings), so not command injection, but paths with leading dashes could be interpreted as flags.
- **Remediation**: Reject paths starting with `-`. Consider canonicalizing and restricting to project directory.

#### [SEC-002] `Regex::new().unwrap()` in Production Code — LOW

- **Location**: `ecc-domain/src/detection/package_manager.rs:112, 124`
- **Evidence**: Panics if regex invalid. Patterns are constants, so safe in practice.
- **Remediation**: Use `LazyLock<Regex>` as done in `session/manager.rs`.

#### [SEC-003] `Regex::new().unwrap()` in `session/manager.rs` — LOW

- **Location**: `ecc-domain/src/session/manager.rs:103-135`
- **Remediation**: Replace `.unwrap()` with `.expect("BUG: invalid regex constant")`.

#### [SEC-004] Transcript Path From Stdin Without Path Restriction — LOW

- **Location**: `ecc-app/src/hook/handlers/tier3_session/mod.rs:107-110`
- **Evidence**: Reads arbitrary file from user-provided path. Acceptable for CLI trust model.

#### [SEC-005] Error Messages Expose Filesystem Paths — LOW

- **Evidence**: Acceptable for CLI tool. Document trust boundary.

#### Positive Security Findings

- No hardcoded secrets
- No `unsafe` blocks
- 1MB stdin size limit enforced
- Session name validation prevents path traversal (`../etc/passwd` rejected)
- Shell commands use safe `Command::new()` pattern (no `sh -c`)
- Script name/args validation with character whitelists

### Test Architecture (Grade: B-)

**Test counts**: 937 tests, all passing. 457 domain (pure unit), ~462 app (impure unit via port doubles), 3 integration, 2 infra, 13 test-support. Pure unit ratio: 97%.

#### [TEST-001] `tier1_simple/handlers.rs` — 445 Lines, Zero Tests — HIGH

- **Location**: `ecc-app/src/hook/handlers/tier1_simple/handlers.rs`

#### [TEST-002] `merge/prompt.rs` — Interactive Prompt Logic Untested — HIGH

- **Location**: `ecc-app/src/merge/prompt.rs` (131 lines)
- **Evidence**: SmartMerge branch with fallback-to-accept on failure. `BufferedTerminal::with_input()` exists for testing but is unused here.
- **Remediation**: Use `BufferedTerminal::with_input("a")` / `with_input("s")` to cover each `ReviewChoice` arm.

#### [TEST-003] `install/mod.rs` and `merge/mod.rs` — Orchestration Untested — HIGH

- **Location**: `ecc-app/src/install/mod.rs` (381 lines), `ecc-app/src/merge/mod.rs` (317 lines)
- **Evidence**: Top-level entry points (`install_global`, `merge_directory`) with zero tests. Helper modules have tests but orchestration logic is uncovered.

#### [TEST-004] `tier3_session` — High Setup-to-Assertion Ratio — MEDIUM

- **Location**: `ecc-app/src/hook/handlers/tier3_session/helpers.rs`
- **Evidence**: 79 setup lines vs 29 assertion lines (ratio 2.7x). Approaching the 3x warning threshold.

#### [TEST-005] `validate.rs` Output Untestable — MEDIUM

- **Location**: `ecc-app/src/validate.rs`
- **Evidence**: 36 `eprintln!`/`println!` calls. Tests can only assert on `bool` return, not output content.

#### [TEST-007] No Property-Based Tests — MEDIUM

- **Evidence**: Zero usage of `proptest`/`quickcheck`. Parsing, merge, and diff algorithms would benefit.

#### [TEST-006] ecc-infra Uses Real Filesystem — LOW

- **Location**: `ecc-infra/src/rustyline_input.rs:71-82`
- **Remediation**: Use `tempfile` crate for automatic cleanup.

#### [TEST-008] No Mutation Testing — LOW

#### [TEST-009] ecc-ports Has Zero Tests — LOW

### Error Handling (Grade: C+)

#### [ERR-001] Silent Swallow of Session Write Failures — CRITICAL

- **Location**: `ecc-app/src/hook/handlers/tier3_session/mod.rs:168, 195, 220, 231`
- **Evidence**: Four `let _ = ports.fs.write(...)` calls discard `FsError` on session persistence, compaction log, and compaction state.
- **Risk**: Session ends with no persisted summary and no user notification. Compaction state loss is unrecoverable.
- **Remediation**: Return warning in `HookResult.stderr` when writes fail.

#### [ERR-002] Silent Swallow of Manifest Write — HIGH

- **Location**: `ecc-app/src/install/mod.rs:265`
- **Evidence**: `let _ = write_manifest(...)` — sole manifest persistence call.
- **Risk**: Future installs treat full update as first-time install, potentially overwriting user files.
- **Remediation**: Add to `InstallSummary.errors`, set `success = false`.

#### [ERR-003] Silent Swallow of Gitignore Write — HIGH

- **Location**: `ecc-app/src/config/gitignore.rs:63`
- **Evidence**: `let _ = fs.write(...)` in a function that reports "Added N gitignore entries" even on failure.
- **Remediation**: Add `error: Option<String>` to `GitignoreResult`.

#### [ERR-004] Silent Swallow of Formatter Failures — HIGH

- **Location**: `ecc-app/src/hook/handlers/tier2_tools.rs:77-91, 227`
- **Evidence**: `let _ = ports.shell.run_command_in_dir(...)` for biome, prettier, gofmt.
- **Remediation**: Populate `HookResult.stderr` with formatter error output.

#### [ERR-005] No Application-Level Error Type — HIGH

- **Location**: All `ecc-app` public use-case functions
- **Evidence**: Port types (`FsError`, `ShellError`) are fully typed, but `ecc-app` uses `anyhow::Result<()>` everywhere. CLI cannot distinguish error categories.
- **Remediation**: Define structured `AppError` enum in `ecc-app`.

#### [ERR-006] `.ok()?` on Settings Write Path — HIGH

- **Location**: `ecc-app/src/install/helpers.rs:138-139`
- **Evidence**: `serde_json::to_string_pretty(&settings).ok()?` converts serialization/write failure to `None`, treated as "nothing to do".
- **Remediation**: Return `Result` and surface error through `InstallSummary.errors`.

#### [ERR-007] `write_session_content` Returns `bool` — MEDIUM

- **Location**: `ecc-app/src/session/mod.rs:163-177`
- **Evidence**: `fs.write(...).is_ok()` discards `FsError` detail.
- **Remediation**: Return `Result<(), FsError>`.

#### [ERR-008] `validate_paths` Uses `.unwrap_or_default()` — MEDIUM

- **Location**: `ecc-app/src/validate.rs:395`
- **Evidence**: `read_dir_recursive(...).unwrap_or_default()` — validation passes on unreadable directory.
- **Remediation**: Report error and return `false`.

#### [ERR-009] `expect()` on Manifest Serialization — MEDIUM

- **Location**: `ecc-app/src/config/manifest.rs:26`
- **Evidence**: `serde_json::to_string_pretty(manifest).expect(...)` — panic risk with future schema changes.
- **Remediation**: Map to `FsError` and propagate.

#### [ERR-010] Audit CLI Falls Back to `.` Silently — MEDIUM

- **Location**: `ecc-cli/src/commands/audit.rs:22`
- **Evidence**: `current_dir().unwrap_or_else(|| PathBuf::from("."))` — wrong directory, no warning.
- **Remediation**: Emit warning or fail early.

#### [ERR-011] REPL Discards `read_line` Errors — MEDIUM

- **Location**: `ecc-app/src/claw/mod.rs:107-111`
- **Evidence**: `Err(_) => break` — identical to EOF. Silent session loss.
- **Remediation**: Log error to stderr before breaking.

#### [ERR-012] `read_json_safe` Conflates Missing and Corrupt — MEDIUM

- **Location**: `ecc-app/src/config/audit/mod.rs:12-18`
- **Evidence**: Both "file missing" and "file corrupt" return `None`.
- **Remediation**: Return `Result<Option<Value>, String>`.

#### [ERR-013] Install Helpers Absorb Parse Failure — MEDIUM

- **Location**: `ecc-app/src/install/helpers.rs:109`
- **Evidence**: Malformed `settings.json` silently prevents deny rules from being applied.
- **Remediation**: Return `Err` on parse failure.

#### [ERR-014] Claw Dispatch Discards `run_repl` Errors — LOW

- **Location**: `ecc-app/src/claw/mod.rs:197, 213, 248, 265`
- **Evidence**: `let _ = run_repl(...)` — CLI exits 0 on error.
- **Remediation**: Propagate or log error, set non-zero exit code.

## Phase 3: Cross-Domain Correlations

Cross-domain analysis reveals 5 escalated risk clusters where findings from different audit domains compound each other.

### [XDOM-001] Untested High-Complexity Hotspot — CRITICAL

**Combined findings**: EVOLUTION-004 + TEST-001

- **File**: `ecc-app/src/hook/handlers/tier1_simple/handlers.rs`
- **Evidence**: Highest complexity density in the codebase (0.144 — 1 branching keyword per 7 lines), 445 lines, 13 public functions, **zero tests**.
- **Risk**: When this file inevitably needs modification (bug fix or new hook), the high branching density combined with zero test coverage maximizes regression risk. The bus factor of 1 means no one else can verify correctness.
- **Action**: Write tests for this file as the highest priority remediation item.

### [XDOM-002] Growing Hotspot With Port Bypass and Untestable Output — CRITICAL

**Combined findings**: EVOLUTION (hotspot score 0.212, growing) + CONV-001 + OBS-003 + TEST-005

- **File**: `ecc-app/src/validate.rs`
- **Evidence**: Second-highest hotspot score, actively growing (578 → 597 lines), bypasses the `TerminalIO` port with 30+ direct `println!`/`eprintln!` calls, output cannot be tested. The only growing hotspot in the codebase that also violates the port convention.
- **Risk**: Every change to this file adds untestable output code that diverges further from the architectural convention.
- **Action**: Refactor to accept `&dyn TerminalIO` before the file grows further.

### [XDOM-003] Silent Write Failures + No Logging = Invisible Data Loss — CRITICAL

**Combined findings**: ERR-001 + ERR-002 + ERR-003 + OBS-001 + OBS-002

- **Files**: `tier3_session/mod.rs`, `install/mod.rs`, `config/gitignore.rs`
- **Evidence**: 9 `let _ =` sites on write paths (session state, manifest, gitignore) combined with zero logging framework. Write failures produce no output whatsoever — no stderr, no log, no return value.
- **Risk**: Users lose session data, install manifests, or gitignore entries with zero indication. The install command reports success while manifest write fails. This is the most operationally dangerous cluster.
- **Action**: Add logging framework (OBS-001), then surface write failures through either logs or `HookResult.stderr`.

### [XDOM-004] Untested Orchestration + Silent Errors = Undetectable Install Failures — CRITICAL

**Combined findings**: TEST-003 + ERR-002 + ERR-003 + ERR-006

- **Files**: `install/mod.rs` (381 lines, zero tests), `install/helpers.rs`, `config/gitignore.rs`
- **Evidence**: The install orchestration layer has zero tests AND contains 3 silent error swallows (manifest write, gitignore write, settings serialization). No test verifies the orchestration logic, and no error is surfaced when it fails.
- **Risk**: A regression in install sequencing or a permission error during install produces no test failure and no user-visible error.
- **Action**: Add orchestration-level tests, then fix error handling.

### [XDOM-005] User-Controlled Paths + Silent Formatter Failures = Blind Execution — CRITICAL

**Combined findings**: SEC-001 + ERR-004

- **Files**: `ecc-app/src/hook/handlers/tier2_tools.rs`
- **Evidence**: File paths from JSON stdin are passed to external formatters (biome, prettier, gofmt) and all results are discarded with `let _ =`. If a malformed path causes a formatter to fail (or a leading-dash path is interpreted as a flag), no diagnostic is produced.
- **Risk**: Combined security + observability gap. A compromised or malformed hook payload triggers unexpected formatter behavior with zero visibility.
- **Action**: Validate file paths (reject leading dashes, canonicalize) and surface formatter exit codes.

## Phase 4: Top 5 Recommendations

Ranked by risk reduction impact, considering cross-domain correlations.

### 1. Add Logging Framework and Surface Write Failures

**Addresses**: OBS-001, OBS-002, ERR-001, ERR-002, ERR-003, ERR-004, ERR-006, XDOM-003
**Effort**: Medium
**Impact**: Eliminates the most dangerous cluster — invisible data loss across session management and installation.

Steps:
1. Add `log = "0.4"` to `ecc-app/Cargo.toml` and `env_logger` to `ecc-cli/Cargo.toml`
2. Initialize logger in `ecc-cli/src/main.rs` (gated on `RUST_LOG`)
3. Replace all `let _ = fs.write(...)` on critical paths with error logging + user notification
4. Add `--verbose`/`-v` flag to top-level CLI (OBS-004)

### 2. Test the Four Untested Orchestration Modules

**Addresses**: TEST-001, TEST-002, TEST-003, XDOM-001, XDOM-004
**Effort**: Medium
**Impact**: Covers 1,274 lines of currently untested code including the highest-complexity file in the codebase.

Priority order:
1. `tier1_simple/handlers.rs` (445 lines, highest complexity, XDOM-001)
2. `install/mod.rs` (381 lines, silent errors, XDOM-004)
3. `merge/mod.rs` (317 lines)
4. `merge/prompt.rs` (131 lines)

### 3. Refactor `validate.rs` to Use `TerminalIO` Port

**Addresses**: CONV-001, OBS-003, TEST-005, XDOM-002
**Effort**: Small
**Impact**: Brings the only non-conforming use case into alignment with the port convention. Enables output testing for 36 message sites.

Steps:
1. Add `terminal: &dyn TerminalIO` parameter to `run_validate` and all private helpers
2. Replace `println!` → `terminal.stdout_write()`, `eprintln!` → `terminal.stderr_write()`
3. Wire `StdTerminal` at CLI adapter
4. Add output assertions to existing tests

### 4. Define a Structured `AppError` Type

**Addresses**: ERR-005, ERR-007, ERR-014
**Effort**: Medium
**Impact**: Enables the CLI to provide differentiated error messages and exit codes. Root cause fix for stringly-typed error propagation.

Steps:
1. Define `AppError` enum in `ecc-app` with variants for `Fs(FsError)`, `Shell(ShellError)`, `Config(String)`, `Logic(String)`
2. Replace `anyhow::Result<()>` in public use-case functions
3. Map `AppError` to user-friendly messages and exit codes at CLI boundary

### 5. Consolidate Duplicated Constants and Parsers

**Addresses**: CONV-002, CONV-003, CONV-006, ARCH-002
**Effort**: Small
**Impact**: Eliminates 3 duplication clusters (package identifiers, frontmatter parsers, stdin limit) that create divergence risk.

Steps:
1. Extract `ECC_PACKAGE_IDENTIFIERS` to `ecc-domain/src/config/mod.rs`
2. Consolidate 3 frontmatter parsers into single `parse_frontmatter` with BOM handling
3. Remove `MAX_STDIN` from CLI, rely on `ecc-app`'s `truncate_stdin`

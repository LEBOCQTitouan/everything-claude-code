#### [TESTING-001] Zero Tests in `ecc-cli` and `ecc-ports` Crates

- **Severity**: HIGH
- **Location**: `crates/ecc-cli/src/` (11 files, 0 tests), `crates/ecc-ports/src/` (6 files, 0 tests)
- **Principle**: Test coverage (80% minimum)
- **Evidence**: `ecc-cli` has 11 source files with zero `#[test]` attributes. `ecc-ports` has 6 source files with zero tests. Combined, these crates represent the CLI entry point and all port trait definitions.

#### [OBSERVABILITY-001] Logging Limited to `ecc-app` Crate Only

- **Severity**: HIGH
- **Location**: `crates/ecc-domain/`, `crates/ecc-ports/`, `crates/ecc-infra/`, `crates/ecc-cli/`
- **Principle**: Operational visibility / structured logging
- **Evidence**: `log` crate is only a dependency of `ecc-app` and `ecc-cli`. The `ecc-domain` and `ecc-ports` crates have zero logging. `ecc-infra` adapters (which perform actual I/O) have no logging at all. All 30+ `log::warn!` calls are concentrated in `ecc-app/src/hook/` and `ecc-app/src/install/`.
- **Risk**: Infrastructure failures (filesystem errors, process execution failures) are invisible. Domain logic errors produce no diagnostic output. Only hook handlers and install logic have any observability.
- **Remediation**: Add `log` as a dependency to `ecc-infra` for adapter-level logging (file I/O failures, process spawning). Consider adding `tracing` for structured spans across the call chain. Domain crate should remain log-free (pure logic).
- **Cross-refs**: [ERROR-001] (silent error swallowing is worse without logging)
- **Delta from 2026-03-14**: IMPROVED (was grade C with 0 logging framework; now `log` is integrated but limited to `ecc-app`)

#### [ERROR-001] Systematic Silent Error Swallowing via `let _ =`

- **Severity**: HIGH
- **Location**: 24 `let _ =` sites across `ecc-app/src/` and `ecc-infra/src/`
- **Principle**: Error handling / fail-fast
- **Evidence**: 24 instances of `let _ =` discarding `Result` values. Key locations:
  - `ecc-infra/src/std_terminal.rs:9-16` — stdout/stderr write failures silenced
  - `ecc-infra/src/rustyline_input.rs:25-81` — history file operations silenced
  - `ecc-app/src/hook/handlers/tier2_notify.rs:20-36` — notification failures silenced
  - `ecc-app/src/claw/handlers/session.rs:25-73` — session save/clear failures silenced
  - `ecc-app/src/install/mod.rs:408` — post-install command failure silenced
- **Risk**: Silent data loss. Session state may fail to persist, notifications may never arrive, and terminal output may be lost — all without any error signal. When combined with limited logging, these failures are invisible.
- **Remediation**: Replace `let _ =` with `if let Err(e) = ...` and log the error. For truly fire-and-forget operations, add a `// intentionally ignored: <reason>` comment. Use clippy's `#[allow(unused_must_use)]` with justification rather than `let _ =`.
- **Cross-refs**: [OBSERVABILITY-001] (no logging in infra makes these truly invisible)
- **Delta from 2026-03-14**: PERSISTENT (was 14 sites, now 24 — count increased with new code)

### MEDIUM

#### [ARCH-001] `ecc-app` Crate Dominates Codebase (60%+ of All Rust Code)

- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/` (51 files, ~17,000 lines — estimated 58% of Rust LOC)
- **Principle**: Single Responsibility / module cohesion
- **Evidence**: `ecc-app` contains 51 of 127 source files. It hosts 623 of 1,193 tests (52%). The crate handles hook dispatch, session management, REPL/claw, installation, config audit, detection, validation, merging, and dev mode — at least 9 distinct bounded contexts.
- **Risk**: Compile times grow linearly. A change in hook handling recompiles session management. Cognitive load is high for any contributor navigating the crate.
- **Remediation**: Extract `claw/`, `hook/`, and `session/` into standalone crates. Each has clear boundaries and minimal cross-dependencies.
- **Delta from 2026-03-14**: PERSISTENT (minor growth from new `dev.rs` module)

#### [ARCH-002] `SystemTime::now()` in Application Layer Bypasses Port Abstraction

- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/src/hook/handlers/tier3_session/mod.rs:25`, `crates/ecc-app/src/claw/handlers/session.rs:13`
- **Principle**: Dependency Inversion / port abstraction
- **Evidence**: Two call sites use `std::time::SystemTime::now()` directly in the application layer. Time is an I/O dependency that should be abstracted behind a port trait (like `Environment` already abstracts `env::var`).
- **Risk**: These modules cannot be deterministically tested for time-dependent behavior. Session timestamps and cost tracking depend on wall clock time.
- **Remediation**: Add a `fn now(&self) -> u64` method to the `Environment` port trait. Inject test-controlled time in tests via `MockEnvironment`.
- **Delta from 2026-03-14**: PERSISTENT

#### [TESTING-002] Heavy Concentration of `unwrap()` in Non-Test Code

- **Severity**: MEDIUM
- **Location**: 284 total `.unwrap()` calls across 39 files
- **Principle**: Defensive programming / graceful error handling
- **Evidence**: While most `unwrap()` calls are in test code (expected), there are `unwrap()` calls in production paths: `session/aliases.rs` (20), `session/mod.rs` (16), `install/mod.rs` (7), `install/helpers.rs` (18), `hook/handlers/tier3_session/mod.rs` (12). Manual inspection shows some are in test modules (`#[cfg(test)]`), but the count is high enough to warrant a targeted review.
- **Risk**: Panics in production on unexpected input. CLI tools should never panic — they should return meaningful error messages.
- **Remediation**: Audit each `unwrap()` in non-test code. Replace with `?` operator, `.unwrap_or_default()`, or `.context("explanation")?` using anyhow.
- **Cross-refs**: [ERROR-001] (error handling philosophy)

#### [TESTING-003] `ecc-infra` Has Only 2 Tests for 6 Source Files

- **Severity**: MEDIUM
- **Location**: `crates/ecc-infra/src/` (6 files, 2 tests)
- **Principle**: Test coverage (80% minimum)
- **Evidence**: Infrastructure adapters (`OsFileSystem`, `ProcessExecutor`, `StdTerminal`, `OsEnvironment`, `RustylineInput`) have minimal testing. Only `rustyline_input.rs` has a `mod tests` block.
- **Risk**: Adapter bugs (incorrect path handling, process spawning edge cases) surface only in integration tests or production.
- **Remediation**: Add unit tests for `OsFileSystem` (at minimum: read/write/exists on temp dirs), `ProcessExecutor` (command success/failure), and `StdTerminal` (output capture).
- **Delta from 2026-03-14**: PERSISTENT

#### [ERROR-002] `.ok()` Used to Silently Convert Errors

- **Severity**: MEDIUM
- **Location**: 5 instances across `ecc-app/src/`
- **Principle**: Error visibility
- **Evidence**: `.ok()` converts `Result` to `Option`, silently discarding the error variant. Used in `session/mod.rs:133` (session file read), `hook/handlers/tier1_simple/dev_hooks.rs:147` (JSON parse), `hook/handlers/tier3_session/mod.rs:663` (JSON parse). In test helpers: `ecc_root_integration.rs:73,97` (acceptable).
- **Risk**: Parse failures and file read errors are silently converted to `None`, making debugging difficult.
- **Remediation**: Use `.map_err(|e| log::warn!(...)).ok()` to at least log the error before discarding.

#### [ERROR-003] Inconsistent Error Taxonomy Across Crates

- **Severity**: MEDIUM
- **Location**: `ecc-ports/src/` (3 error enums), `ecc-app/src/` (uses `anyhow::Result`), `ecc-domain/src/` (returns `String` errors)
- **Principle**: Consistent error boundary translation
- **Evidence**: `ecc-ports` defines typed errors (`FsError`, `ShellError`, `TerminalError`) using `thiserror`. `ecc-app` uses `anyhow::Result` (erased types). `ecc-domain` returns `Result<T, String>` in several functions (e.g., `package_manager.rs` validation). Three different error philosophies in three layers.
- **Risk**: Error context is lost at layer boundaries. A `FsError::NotFound` from ports becomes an opaque `anyhow::Error` in app, losing the ability to pattern-match on error types.
- **Remediation**: Standardize on `thiserror` for typed errors at each layer boundary. Use `anyhow` only at the CLI entry point. Domain should define its own error types rather than using `String`.

#### [OBSERVABILITY-002] No Structured Logging or Correlation IDs

- **Severity**: MEDIUM
- **Location**: All `log::warn!` calls in `crates/ecc-app/src/`
- **Principle**: Structured observability
- **Evidence**: All logging uses unstructured `log::warn!` with format strings. No structured fields, no correlation IDs, no span context. For a CLI tool this is acceptable, but as the hook system processes events in sequence, correlating which hook produced which warning is difficult.
- **Risk**: Debugging hook chains requires manual log reading. No ability to filter or aggregate by hook ID, session, or event type.
- **Remediation**: Consider `tracing` with structured fields: `warn!(hook_id = %id, session = %session, "failed to write")`. Low priority for a CLI tool but valuable as the hook system grows.

#### [OBSERVABILITY-003] No Health Check or Self-Diagnostic for Hook System

- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/src/hook/`
- **Principle**: Operational readiness
- **Evidence**: The hook dispatch system (`dispatch()`, 83 lines) processes hooks silently. There is no dry-run mode, no hook health check, and no way to verify the hook pipeline is correctly configured without actually triggering hooks.
- **Risk**: Hook misconfiguration (wrong IDs, missing handlers) is only discovered at runtime.
- **Remediation**: The existing `ecc audit` command partially addresses this. Consider adding a `--dry-run` flag to `ecc hook` for testing hook configurations.

#### [CONVENTIONS-001] Duplicate Module Names Across Crates

- **Severity**: MEDIUM
- **Location**: `mod.rs` (20 files), `helpers.rs` (4 files), `validate.rs` (3 files), `detect.rs` (3 files), `audit.rs` (3 files)
- **Principle**: Naming clarity / discoverability
- **Evidence**: 20 files named `mod.rs` is unavoidable in Rust's module system. However, 4 files named `helpers.rs` across different crates (`ecc-app/src/install/helpers.rs`, `ecc-app/src/merge/helpers.rs`, `ecc-app/src/hook/handlers/tier1_simple/helpers.rs`, `ecc-app/src/hook/handlers/tier3_session/helpers.rs`) reduce discoverability.
- **Risk**: Navigating to "the right helpers.rs" requires full path context. Search results are ambiguous.
- **Remediation**: Rename to domain-specific names: `install_io.rs`, `merge_transforms.rs`, `hook_formatting.rs`, `session_parsing.rs`.

## Cross-Domain Correlations

### Escalation 1: Bus Factor + Hotspot (CRITICAL)

- **Signal**: [EVOLUTION-001] Bus factor = 1 + [EVOLUTION-002] tier3_session/mod.rs is #1 hotspot
- **Compound risk**: The most-changed, most-complex file in the codebase (1,122 lines, 11 commits, 7 functions >50 lines) is maintained by a single contributor. Any disruption to that contributor blocks all development on the hook system.
- **Action**: Highest priority for knowledge sharing and pair programming.

### Escalation 2: Hotspot + Oversized + Untested CLI (HIGH)

- **Signal**: [EVOLUTION-002] oversized files + [TESTING-001] zero CLI tests
- **Compound risk**: The CLI layer that wires everything together has zero tests, while the application layer it depends on has files exceeding size limits and growing complexity. A CLI change could silently break argument parsing or subcommand routing with no safety net.
- **Action**: Add CLI smoke tests using `Cli::try_parse_from()`.

### Escalation 3: Error Swallowing + No Infra Logging (HIGH)

- **Signal**: [ERROR-001] 24 `let _ =` sites + [OBSERVABILITY-001] no logging in `ecc-infra`
- **Compound risk**: Infrastructure adapters (`OsFileSystem`, `ProcessExecutor`, `StdTerminal`) silently discard errors and have no logging. Failures in the I/O layer are completely invisible. Combined with only 2 tests in `ecc-infra`, the adapter layer is a blind spot.
- **Action**: Add `log` dependency to `ecc-infra`. Replace `let _ =` with logged error handling.

### Escalation 4: Error Taxonomy + Port Boundaries (MEDIUM)

- **Signal**: [ERROR-003] inconsistent error types + [ARCH-002] port boundary violations
- **Compound risk**: Three different error philosophies (typed `thiserror`, erased `anyhow`, raw `String`) mean error context is lost when crossing layer boundaries. The `SystemTime::now()` port bypass shows the boundary discipline is not 100%.
- **Action**: Standardize error handling at each architectural layer.

## Top 5 Recommendations

1. **Split `tier3_session/mod.rs` (1,122 lines)** — Extract each session hook into its own file. This is the #1 hotspot, the largest file, and contains 7 functions over 50 lines. Splitting reduces merge conflicts, improves test granularity, and makes the code navigable.

Combined with zero logging in the infrastructure layer, I/O failures are completely invisible. Add `log` to `ecc-infra/Cargo.toml` and replace `let _ = expr` with `if let Err(e) = expr { log::warn!("...") }`.

3. **Add CLI smoke tests for `ecc-cli`** — Zero tests for the entry point. Use `Cli::try_parse_from(&["ecc", "version"])` to verify argument parsing. This takes <30 minutes and prevents silent CLI regressions.

4. **Abstract `SystemTime::now()` behind the `Environment` port** — Two direct time access calls in the application layer bypass the port abstraction. Add `fn now_epoch_secs(&self) -> u64` to the `Environment` trait. This enables deterministic testing of time-dependent logic.

5. **Onboard a second Rust contributor** — Bus factor = 1 across 29K lines is the single biggest organizational risk. Start with pair programming on `ecc-domain` (pure logic, easiest onramp) and create "good first issue" tasks for the detection module.

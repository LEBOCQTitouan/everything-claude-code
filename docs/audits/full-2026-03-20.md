## Executive Summary

The codebase demonstrates **strong architectural discipline** with a well-implemented hexagonal architecture. Dependency direction is flawless -- no inner-layer imports of outer-layer code. The domain crate is pure (zero I/O imports). Port traits are well-defined with matching adapter implementations and test doubles.

The primary risks center on **bus factor** (single contributor to all Rust code), **file size violations** (3 files exceed the 800-line limit), **sparse observability** (only `log::warn` usage, no structured logging or metrics), and **missing test coverage** in the infra and CLI layers.

All 1,141 tests pass. Clippy reports zero warnings. No hardcoded secrets detected. No unsafe code blocks.

## Findings

### CRITICAL

#### [CORR-001] Bus Factor 1 Across Entire Rust Codebase
- **Severity**: CRITICAL (escalated from HIGH via cross-domain correlation: bus factor + hotspot)
- **Location**: All files under `crates/`
- **Principle**: Organizational resilience
- **Evidence**: `git log --format='%aN' --since="2025-09-20" -- '*.rs' | sort -u` returns a single contributor (titouanlebocq). 100% of 903 commits in the 180-day window touching Rust code are from one person. Every hotspot file (top 20 by churn) has bus factor = 1.
- **Risk**: If the sole contributor becomes unavailable, no one can maintain the core CLI. No knowledge transfer path exists for the Rust implementation.
- **Remediation**: Establish contributor onboarding documentation for the Rust workspace. Pair program on complex modules (tier3_session, install/helpers). Consider recording architecture decision records for non-obvious design choices.
- **Cross-refs**: [EVO-001], [EVO-003]

### HIGH

#### [ARCH-001] Three Files Exceed 800-Line Limit
- **Severity**: HIGH
- **Location**: `crates/ecc-app/src/hook/handlers/tier3_session/mod.rs` (1,122 lines), `crates/ecc-app/src/install/helpers.rs` (986 lines), `crates/ecc-app/src/config/audit/checks.rs` (816 lines)
- **Principle**: Single Responsibility, file size limit (< 800 lines per coding-style.md)
- **Evidence**: Three files exceed the documented 800-line maximum. The tier3_session handler alone is 1,122 lines -- 40% over the limit.
- **Risk**: Large files resist comprehension and increase merge conflict probability. The tier3_session handler is also the 6th-highest churn file.
- **Remediation**: Extract logical sub-handlers from tier3_session/mod.rs (as was done for tier1_simple). Split install/helpers.rs into focused helper modules (settings, statusline, file operations). Break checks.rs by check category.
- **Cross-refs**: [CONV-001], [EVO-002]

#### [TEST-001] Zero Unit Tests in Infrastructure and CLI Layers
- **Severity**: HIGH
- **Location**: `crates/ecc-infra/src/` (2 tests total), `crates/ecc-cli/src/` (0 tests), `crates/ecc-ports/` (0 tests)
- **Principle**: 80% test coverage requirement
- **Evidence**: ecc-infra has 2 tests (in rustyline_input.rs only). ecc-cli has 0 inline tests. ecc-ports has 0 tests. All 9 CLI command handlers (`hook.rs`, `audit.rs`, `validate.rs`, etc.) are untested at the unit level. Integration tests exist (5 tests in ecc-integration-tests) but provide minimal coverage.
- **Risk**: CLI argument handling, adapter wiring, and error translation at the boundary are unverified. Regressions in the entry point layer go undetected.
- **Remediation**: Add unit tests for CLI command handlers using test doubles from ecc-test-support. Add tests for OsFileSystem, ProcessExecutor, and OsEnvironment edge cases. Target 80% line coverage per crate.
- **Cross-refs**: [ERR-002], [OBS-001]

#### [ERR-001] App Layer Uses anyhow Exclusively -- No Domain Error Types
- **Severity**: HIGH
- **Location**: `crates/ecc-app/src/` (entire crate)
- **Principle**: Error boundary translation, domain-driven error taxonomy
- **Evidence**: The app layer returns `anyhow::Result` from all public functions. The ports layer defines 3 well-typed error enums (FsError, ShellError, TerminalError), but the app layer does not define its own error types. It wraps port errors with `anyhow!` context strings instead of translating them into domain-meaningful errors.
- **Risk**: Callers cannot programmatically match on error variants. Error messages from different subsystems are indistinguishable. Makes retry logic and graceful degradation impossible.
- **Remediation**: Define an `AppError` enum in ecc-app with variants like `ConfigNotFound`, `InstallFailed`, `HookDispatchError`, etc. Map port errors to app errors at boundaries. Reserve anyhow for truly unexpected failures.
- **Cross-refs**: [OBS-002], [TEST-001]

#### [OBS-001] No Structured Logging, Metrics, or Tracing
- **Severity**: HIGH
- **Location**: `crates/ecc-app/src/` (17 log statements, all `log::warn`)
- **Principle**: Observability at application boundaries
- **Evidence**: The entire codebase uses only `log::warn` (17 occurrences, concentrated in install/helpers.rs). No `log::info`, `log::error`, `log::debug`, or `log::trace` calls exist in the app layer. No structured logging (key=value pairs). No metrics collection. No request/operation tracing. No health check endpoint.
- **Risk**: Debugging production issues requires adding temporary logging. No visibility into CLI usage patterns, error rates, or performance characteristics.
- **Remediation**: Add `log::info` at use case entry/exit points. Add `log::error` for all error paths. Consider structured logging with `tracing` crate for richer context. Add `log::debug` for configuration resolution and hook dispatch decisions.
- **Cross-refs**: [ERR-001], [ERR-003]

#### [EVO-001] Extreme Churn Concentration in App Layer
- **Severity**: HIGH
- **Location**: `crates/ecc-app/src/config.rs` (4,932 churn), `crates/ecc-app/src/hook/handlers/` (3,500+ churn combined)
- **Principle**: Stability principle -- frequently changing modules should be unstable (high I, low Ca)
- **Evidence**: Top 5 churn files are all in ecc-app. config.rs alone has 4,932 lines of churn (additions + deletions) over 180 days. The hook handler subsystem (tier1_simple, tier2_tools, tier3_session) accounts for 5,000+ combined churn. Meanwhile, ecc-domain files are comparatively stable.
- **Risk**: Concentrated instability suggests the app layer absorbs too many responsibilities. High churn files correlate with higher defect density.
- **Remediation**: Audit config.rs for responsibilities that belong in domain (pure logic) vs app (orchestration). Consider whether hook handler tiers warrant their own crate or at least clearer module boundaries.
- **Cross-refs**: [ARCH-001], [CORR-001]

### MEDIUM

#### [TEST-002] Five Files with Nesting Depth Exceeding 6 Levels
- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/src/merge/mod.rs` (depth 9), `crates/ecc-domain/src/config/validate.rs` (depth 7), `crates/ecc-app/src/validate.rs` (depth 7), `crates/ecc-app/src/merge/prompt.rs` (depth 7), `crates/ecc-app/src/claw/handlers/session.rs` (depth 7)
- **Principle**: Nesting depth < 4 levels (coding-style.md)
- **Evidence**: merge/mod.rs reaches 9 levels of brace nesting. Four additional files exceed 6 levels. The coding standard specifies a maximum of 4 levels.
- **Risk**: Deep nesting reduces readability and increases cyclomatic complexity. Correlated with higher defect rates.
- **Remediation**: Apply early-return patterns, extract nested logic into helper functions, and use `match` guard clauses to flatten control flow.
- **Cross-refs**: [ARCH-001]

#### [ERR-002] Production Code Contains unwrap() Calls
- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/src/detection/language.rs` (6 occurrences), `crates/ecc-cli/src/commands/hook.rs` (2 occurrences)
- **Principle**: Never panic in production code
- **Evidence**: 8 `unwrap()` calls exist outside test modules. language.rs has 6 unwraps in what appears to be regex compilation (likely safe but convention-violating). hook.rs has 2 unwraps on iterator operations that could panic on empty collections.
- **Risk**: Runtime panics in CLI tool cause ungraceful exits with no user-friendly error message. The hook.rs unwraps are reachable if argument parsing produces unexpected state.
- **Remediation**: Replace language.rs unwraps with `lazy_static` or `std::sync::OnceLock` for regex compilation. Replace hook.rs unwraps with proper error handling and user-facing messages.
- **Cross-refs**: [ERR-001], [SEC-001]

#### [ERR-003] Swallowed Errors via .ok() in Non-Trivial Operations
- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/src/config/manifest.rs:9-14`, `crates/ecc-app/src/config/detect.rs:38`, `crates/ecc-app/src/config/audit/hooks_diff.rs:17-20`
- **Principle**: Never silently swallow errors
- **Evidence**: Several `.ok()` calls convert Result to Option, discarding error context. In manifest.rs, a failed file read and two failed JSON parses are silently converted to `None`. In hooks_diff.rs, deserialization failures are silently ignored.
- **Risk**: Configuration loading fails silently, producing confusing downstream behavior. Users see "no config found" when the real issue is a parse error.
- **Remediation**: Replace `.ok()` with explicit error handling. Log the error at warn level before converting to None, or propagate the error upward.
- **Cross-refs**: [OBS-001]

#### [OBS-002] Log Statements Concentrated in Single Module
- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/src/install/helpers.rs` (11 of 17 total log::warn calls)
- **Principle**: Consistent observability across all modules
- **Evidence**: 11 out of 17 total log statements are in install/helpers.rs. Entire subsystems (session management, claw, merge, detect, validate, hook dispatch) have zero logging. The imbalance means install issues are debuggable but all other operations are opaque.
- **Risk**: Operational blind spots in hook dispatch, session management, and configuration merge -- the highest-churn subsystems.
- **Remediation**: Add logging to hook dispatch entry/exit, session lifecycle events, configuration merge decisions, and validation results.
- **Cross-refs**: [OBS-001], [EVO-001]

#### [DOC-001] Low Doc Comment Coverage on Public API
- **Severity**: MEDIUM
- **Location**: All crates
- **Principle**: Public API documentation
- **Evidence**: Doc comment coverage: ecc-domain 131/444 (30%), ecc-ports 5/21 (24%), ecc-app 158/316 (50%). Most public structs, enums, and functions lack `///` doc comments. The ports crate -- which defines the architectural contracts -- has only 5 documented items out of 21.
- **Risk**: New contributors cannot understand the purpose of types and functions without reading implementation details. IDE tooltips provide no guidance.
- **Remediation**: Prioritize doc comments on all port traits (they are the architectural contracts), domain types, and app-layer public functions. Target 80% doc coverage.
- **Cross-refs**: [CORR-001]

#### [EVO-002] High Co-Change Coupling in Hook Handlers
- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/src/hook/handlers/` subsystem
- **Principle**: Single Responsibility, module independence
- **Evidence**: Hook handler files (tier1_simple/handlers.rs, tier3_session/mod.rs, tier2_tools.rs) frequently change together. The mod.rs dispatcher, all three tier modules, and their helpers have high co-change rates, suggesting tight implicit coupling.
- **Risk**: Changes to one hook tier ripple into others, increasing the blast radius of modifications.
- **Remediation**: Review whether hook tiers share state or interfaces that could be formalized. Consider a registry pattern to decouple dispatch from handler implementation.
- **Cross-refs**: [ARCH-001], [EVO-001]

#### [CONV-001] Inconsistent Module Decomposition Patterns
- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/src/hook/handlers/tier1_simple/` vs `crates/ecc-app/src/hook/handlers/tier3_session/`
- **Principle**: Consistent patterns across similar modules
- **Evidence**: tier1_simple is well-decomposed into focused sub-modules (clean_craft_hooks.rs, dev_hooks.rs, doc_hooks.rs, git_hooks.rs, meta_hooks.rs, helpers.rs). tier3_session uses a single 1,122-line mod.rs with a separate helpers.rs. tier2_tools is a single 691-line file. The decomposition strategy is inconsistent across tiers.
- **Risk**: Inconsistency increases cognitive load. Developers must learn different organizational patterns for each tier.
- **Remediation**: Apply the tier1_simple decomposition pattern to tier3_session and tier2_tools. Extract handler groups into focused sub-modules.
- **Cross-refs**: [ARCH-001]

### LOW

#### [SEC-001] Input Validation Present but Not Comprehensive
- **Severity**: LOW
- **Location**: `crates/ecc-domain/src/detection/package_manager.rs`
- **Principle**: Validate all external input at boundaries
- **Evidence**: Package manager detection includes command injection prevention (unsafe character validation for script names and arguments). CLI uses Clap for argument parsing (structural validation). However, file content read from disk (CLAUDE.md, hooks.json, settings.json) is parsed with minimal validation beyond JSON/YAML deserialization.
- **Risk**: Malformed configuration files could produce unexpected behavior rather than clear error messages.
- **Remediation**: Add schema validation for hooks.json and settings.json structures after deserialization. Produce user-friendly error messages for malformed config.
- **Cross-refs**: [ERR-003]

#### [CONV-002] Dual Detection Modules
- **Severity**: LOW
- **Location**: `crates/ecc-domain/src/detection/` and `crates/ecc-app/src/detection/`
- **Principle**: Clear module boundaries, avoid duplication
- **Evidence**: Both ecc-domain and ecc-app contain `detection/` modules with language.rs, framework.rs, and package_manager.rs. Domain detection contains pure rules and data structures. App detection contains I/O-dependent detection logic. The naming overlap could confuse contributors.
- **Risk**: Ambiguity about where new detection logic should be placed. Risk of accidentally putting I/O in domain.
- **Remediation**: Consider renaming app/detection to app/detect_service or app/detection_service to distinguish orchestration from rules.
- **Cross-refs**: [DOC-001]

#### [DOC-002] CLAUDE.md Test Count Slightly Outdated
- **Severity**: LOW
- **Location**: `CLAUDE.md`
- **Principle**: Documentation accuracy
- **Evidence**: CLAUDE.md states "1190 tests" but the actual count is 1,141 passing tests (507 + 617 + 2 + 5 + 5 + 5 = 1,141). The gotchas section mentions "currently 1180" which is also outdated.
- **Risk**: Misleading reference for contributors checking test health.
- **Remediation**: Update both test count references in CLAUDE.md to the current passing count.

## Top 5 Recommendations

1. **Establish contributor knowledge transfer for Rust codebase** [CORR-001]. The single-contributor bus factor is the highest organizational risk. Document architecture decisions, pair program on complex modules, and onboard at least one additional contributor to the Rust workspace.

2. **Add unit tests to CLI and infra layers** [TEST-001]. The entry point and adapter layers have near-zero test coverage. Add tests for all 9 CLI command handlers and the 4 infra adapters using existing test doubles. Target 80% coverage per crate.

3. **Decompose oversized files** [ARCH-001]. Split `tier3_session/mod.rs` (1,122 lines), `install/helpers.rs` (986 lines), and `audit/checks.rs` (816 lines) into focused sub-modules following the tier1_simple decomposition pattern.

4. **Add structured logging across all modules** [OBS-001, OBS-002]. Extend logging beyond install/helpers.rs. Add info-level logging to use case entry points, error-level for all error paths, and debug-level for configuration resolution and hook dispatch.

5. **Define app-layer error types** [ERR-001]. Replace `anyhow::Result` in ecc-app's public API with a typed `AppError` enum. Map port-level errors to domain-meaningful variants. This enables programmatic error handling by callers and improves error messages.

## Methodology

- **Architecture**: Analyzed Cargo.toml dependency graphs, `use` statement direction, trait definitions, and implementations across all 7 crates.
- **Evolution**: Git log analysis over 180-day window (2025-09-20 to 2026-03-20). Churn = additions + deletions per file. Bus factor = unique author count per file.
- **Testing**: Counted `#[test]` annotations per crate. Ran `cargo test` to verify all pass. Identified untested source files.
- **Security**: Searched for hardcoded secrets, unsafe blocks, and input validation patterns.
- **Observability**: Grepped for `log::` usage patterns and distribution across modules.
- **Error Handling**: Analyzed error type definitions, `unwrap()`, `.ok()`, `panic!`, and `anyhow` usage patterns.
- **Conventions**: Reviewed naming patterns, module structure consistency, and file size compliance.
- **Documentation**: Counted `///` doc comments relative to `pub` items per crate.
- **Cross-domain**: Applied correlation rules from the audit specification to escalate compound signals.

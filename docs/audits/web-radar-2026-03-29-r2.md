## Techniques

### Adopt

#### Promptware Engineering — Treat Prompts as First-Class Software Artifacts
- **Ring**: Adopt
- **Strategic Fit**: 5/5 | **Maturity**: 4/5 | **Migration Effort**: 1/5
- **Rationale**: ECC already treats agent directives, skills, and commands as versioned Markdown artifacts with frontmatter. The Promptware Engineering framework (arXiv 2503.02400) formalizes this with lifecycle practices (requirements, testing, debugging, evolution). ECC is ahead of the curve — adopting the formal vocabulary and remaining practices (prompt testing, prompt monitoring) strengthens the foundation.
- **Source Citations**: [arXiv 2503.02400](https://arxiv.org/abs/2503.02400), [arXiv 2503.02400v2](https://arxiv.org/html/2503.02400v2), [ADS Abstract](https://ui.adsabs.harvard.edu/abs/2025arXiv250302400C/abstract)

#### Compiler-Enforced Architecture via Crate Boundaries
- **Ring**: Adopt
- **Strategic Fit**: 5/5 | **Maturity**: 5/5 | **Migration Effort**: 0/5
- **Rationale**: ECC already uses this pattern (domain → ports → infra → app → CLI). Multiple 2025-2026 sources confirm Rust's crate visibility as the gold standard for enforcing Clean Architecture at compile time. No migration needed — continue and deepen.
- **Source Citations**: [howtocodeit.com](https://www.howtocodeit.com/guides/master-hexagonal-architecture-in-rust), [barrage.net](https://www.barrage.net/blog/technology/how-to-apply-hexagonal-architecture-to-rust), [LogRocket](https://blog.logrocket.com/best-way-structure-rust-web-services/)

### Trial

#### OARS + Laddering + MECE Socratic Questioning Protocol
- **Ring**: Trial
- **Strategic Fit**: 5/5 | **Maturity**: 3/5 | **Migration Effort**: 2/5
- **Rationale**: Research confirms Socratic questioning (OARS reflections, laddering depth drilling, MECE decomposition) significantly improves requirements elicitation quality. LLM agents achieve above-minimum human competency on motivational interviewing measures. Already captured as BL-098 for grill-me upgrade. Trial because the combination of all 4 techniques in an LLM context is novel.
- **Source Citations**: [arXiv 2407.08095](https://arxiv.org/html/2407.08095v1), [LadderBot KIT](https://publikationen.bibliothek.kit.edu/1000096469/64959479), [Princeton SocraticAI](https://princeton-nlp.github.io/SocraticAI/)

#### Harness Engineering as Competitive Moat
- **Ring**: Trial
- **Strategic Fit**: 5/5 | **Maturity**: 4/5 | **Migration Effort**: 2/5
- **Rationale**: 2026 consensus: the competitive moat is harness quality (hooks, guardrails, deterministic control), not model selection. ECC's phase-gate hooks, state machine enforcement, and atomic commit workflow directly address this. Trial to formalize harness reliability metrics and benchmark against Anthropic's reference harness patterns.
- **Source Citations**: [Anthropic Engineering](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents), [Agent Engineering 2026](https://www.agent-engineering.dev/article/harness-engineering-in-2026-the-discipline-that-makes-ai-agents-production-ready), [Medium](https://aakashgupta.medium.com/2025-was-agents-heres-why-that-changes-everything-073e9877655e)

### Assess

#### Graph-Based Agent Orchestration (LangGraph-style)
- **Ring**: Assess
- **Strategic Fit**: 4/5 | **Maturity**: 3/5 | **Migration Effort**: 4/5
- **Rationale**: Graph-based frameworks enable stateful multi-actor workflows with resumable checkpoints. ECC currently uses linear pipeline (spec → design → implement) with parallel wave dispatch. Worth assessing whether a graph topology would improve complex multi-agent coordination, but migration effort is high.
- **Source Citations**: [StackOne](https://www.stackone.com/blog/ai-agent-tools-landscape-2026/), [Lindy](https://www.lindy.ai/blog/ai-agent-architecture), [Redis](https://redis.io/blog/ai-agent-architecture/)

#### Module-First Over Over-Crating Strategy
- **Ring**: Assess
- **Strategic Fit**: 3/5 | **Maturity**: 4/5 | **Migration Effort**: 3/5
- **Rationale**: 2026 consensus shifts toward using deeply nested modules within fewer crates before extracting. ECC has 9 crates — this may be appropriate for its hexagonal boundaries, but assess whether some crates (e.g., ecc-test-support, ecc-flock) should be modules within existing crates to reduce build overhead.
- **Source Citations**: [matklad](https://matklad.github.io/2021/08/22/large-rust-workspaces.html), [reintech](https://reintech.io/blog/cargo-workspace-best-practices-large-rust-projects), [OneHorizon](https://onehorizon.ai/blog/modern-rust-best-practices-in-2026-beyond-the-borrow-checker)

### Trial

#### sccache + mold for Build Acceleration
- **Ring**: Trial
- **Strategic Fit**: 4/5 | **Maturity**: 4/5 | **Migration Effort**: 1/5
- **Rationale**: sccache caches compilation artifacts (11-14% speedup on test builds). mold linker is fastest for Linux. Combined with Cranelift backend: up to 75% incremental build speedup in real-world projects. Low effort to trial — add to dev environment, not CI.
- **Source Citations**: [Rust Perf Book](https://nnethercote.github.io/perf-book/build-configuration.html), [corrode.dev](https://corrode.dev/blog/tips-for-faster-rust-compile-times/), [benw.is](https://benw.is/posts/how-i-improved-my-rust-compile-times-by-seventy-five-percent)

#### Miri for Unsafe Code Verification
- **Ring**: Trial
- **Strategic Fit**: 3/5 | **Maturity**: 4/5 | **Migration Effort**: 1/5
- **Rationale**: ECC uses `libc` and POSIX flock which involve unsafe boundaries. Miri detects undefined behavior in unsafe code. Low effort to trial on targeted crates (ecc-flock). Not needed project-wide since most crates are pure safe Rust.
- **Source Citations**: [markaicode.com](https://markaicode.com/rust-static-analysis-tools-comparison-2025/), [Trail of Bits](https://blog.trailofbits.com/2021/11/09/write-rust-lints-without-forking-clippy/), [analysis-tools.dev](https://analysis-tools.dev/tool/clippy)

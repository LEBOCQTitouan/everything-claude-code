## Executive Summary

The ECC codebase has improved meaningfully since the March 29 audit. Architectural boundaries remain pristine -- zero dependency rule violations across layers. The typed hook merge path now has 12 tests (up from 0 in the prior audit). The install pipeline was decomposed into focused modules. Clippy has 3 new pedantic lint errors (`is_multiple_of`) that need attention but are trivial fixes.

Primary risks now center on: (1) a failing test in `sqlite_bypass_store` caused by SQL datetime comparison logic, (2) seven files exceeding the 800-line project limit, (3) high `unwrap()`/`expect()` density in domain code (particularly `Regex::new().expect()` scattered through multiple modules), and (4) low doc-comment coverage in the application layer (2.7%). The bus factor remains maximum risk: a single contributor owns 100% of core Rust code.

Security posture is strong. No hardcoded secrets, no `unsafe` blocks in production code, proper port-based I/O abstraction. One SQL formatting concern in `sqlite_bypass_store::prune()` uses string interpolation for a `u64` duration parameter -- not exploitable (integer type prevents injection) but violates the parameterized-query convention.

Observability improved with structured logging in hook dispatch and workflow transitions, but the domain layer has zero logging statements (by design, acceptable for a pure domain) and the infrastructure layer has only 11.

## Cross-Domain Correlations

### [CORR-001] HIGH -- Bus Factor + Entire Codebase Hotspot

- **Severity**: HIGH (escalated from MEDIUM via bus-factor + hotspot rule)
- **Sources**: Evolution + Architecture
- **Signal**: Single contributor owns 100% of 462 Rust source files across 2,652 commits. Every file is a bus-factor risk.
- **Risk**: No knowledge redundancy. Extended absence of the sole contributor halts all development.
- **Remediation**: Document architectural decisions in ADRs (partially done). Expand `docs/ARCHITECTURE.md` with rationale for key design choices. Consider pair-programming or mentorship sessions.

### [CORR-002] MEDIUM -- Oversized Files + High Churn = Rigidity Smell

- **Severity**: MEDIUM
- **Sources**: Evolution + Conventions + Architecture
- **Signal**: `validate/patterns.rs` (2,355 lines, 54 functions) is the most oversized file and receives frequent changes. Combined with `tests_helpers.rs` (2,009 lines), these two files alone account for 4,364 lines -- 4.3% of the codebase in 2 files.
- **Design Smell**: **Rigidity** -- large files with high churn resist change and make reviews harder.
- **Remediation**: Extract `validate/patterns.rs` into per-concern submodules (frontmatter validation, code-block scanning, section validation). Move `tests_helpers.rs` fixture data into test data files.

### [CORR-003] MEDIUM -- Failing Test + SQL Interpolation = Fragility Signal

- **Severity**: MEDIUM
- **Sources**: Testing + Security + Error Handling
- **Signal**: `sqlite_bypass_store::tests::bypass_prune_removes_old` fails because the `prune()` method uses `format!("datetime('now', '-{} days')", older_than_days)` -- SQLite evaluates `datetime('now', ...)` at query time, but the test inserts a record with timestamp `2026-04-06T10:00:00Z` (hardcoded "recent"), which will become "old" as real time progresses. The SQL also uses string interpolation instead of parameterized queries.
- **Risk**: Flaky test that will fail differently depending on when it runs. The SQL interpolation, while not exploitable (u64 input), violates the project's parameterized-query convention.
- **Remediation**: (1) Fix the test to use relative timestamps or inject a clock. (2) Rewrite the `prune()` SQL to use parameterized `datetime()` with bound parameters.

### [CORR-004] LOW -- Convention Divergence + Coupled Modules

- **Severity**: LOW
- **Sources**: Conventions + Architecture
- **Signal**: 48 error enum types across the codebase with inconsistent naming: some use `*AppError` (app layer), some use `*StoreError` (infra layer), and `WorktreeError` is duplicated across two crates. While each follows its layer convention, the cross-crate duplication of `WorktreeError` creates ambiguity.
- **Remediation**: Rename one of the `WorktreeError` types (e.g., `WorktreePortError` in ports, `WorktreeAdapterError` in infra) to disambiguate.

### Testing

#### [TEST-001] Failing Test: bypass_prune_removes_old
- **Severity**: HIGH
- **Location**: `crates/ecc-infra/src/sqlite_bypass_store.rs:221-237`
- **Principle**: All tests must pass (CI red)
- **Evidence**: Test inserts a record with `timestamp = "2026-04-06T10:00:00Z"` as "recent" and expects it to survive a `prune(1)` call. The `prune()` method uses `datetime('now', '-1 days')` which evaluates at query time. As real time progresses beyond 2026-04-07, the "recent" record becomes "old" and gets pruned. Assertion fails: `left: 2, right: 1`.
- **Risk**: CI is red. Blocks all merges if CI is enforced.
- **Remediation**: Inject a `Clock` port into `prune()` or use relative timestamps in the test. Replace `datetime('now', ...)` with a parameterized cutoff timestamp.

### Security

#### [SEC-001] No Hardcoded Secrets
- **Severity**: INFO (positive)

#### [SEC-002] No Unsafe Code in Production

#### [SEC-003] SQL Interpolation in bypass_store::prune()
- **Severity**: MEDIUM
- **Location**: `crates/ecc-infra/src/sqlite_bypass_store.rs:154-160`
- **Principle**: Always use parameterized queries
- **Evidence**: `format!("DELETE FROM bypass_decisions WHERE timestamp < {cutoff}")` where `cutoff = format!("datetime('now', '-{} days')", older_than_days)`. The `older_than_days` is `u64` (not user-controlled string), so this is not exploitable, but it violates the project's own security rule.
- **Risk**: Sets a precedent for string-formatted SQL. Future developers may copy this pattern with string inputs.
- **Remediation**: Rewrite to use parameterized query with bound timestamp.

#### [SEC-004] Path Inputs Not Validated at CLI Boundary
- **Severity**: MEDIUM
- **Location**: `crates/ecc-cli/src/commands/`
- **Principle**: Validate all external input at boundaries
- **Evidence**: CLI commands accept paths from user input without explicit path traversal validation. The port abstraction (`FileSystem` trait) provides safety, but no explicit validation rejects `../../../etc/passwd` style inputs.
- **Risk**: Low -- the `FileSystem` trait operations are scoped, and the CLI is a local tool (not a server). But defense-in-depth recommends boundary validation.
- **Remediation**: Add a `validate_path()` utility for CLI-provided paths.

#### [SEC-005] Dependency: `serde-saphyr` Instead of `serde_yml`
- **Severity**: LOW
- **Location**: `Cargo.toml`
- **Evidence**: Using `serde-saphyr 0.0.22`. Prior audit flagged `serde_yml` with quality concerns. This appears to be a resolved finding -- the dependency was migrated.

#### [CONV-003] Regex Compilation Pattern Inconsistency
- **Severity**: MEDIUM
- **Location**: Multiple domain modules
- **Principle**: Consistent patterns across codebase
- **Evidence**: Three patterns coexist:
  1. `LazyLock<Regex>` with `expect()` (app/detection, preferred)
  2. `OnceLock` with `get_or_init` (domain/audit_web)
  3. Inline `Regex::new().expect()` per call (domain/spec, domain/memory)
- **Risk**: Inconsistent performance characteristics. Inline compilation runs on every call.
- **Remediation**: Standardize on `LazyLock<Regex>` across all crates. Extract shared regex patterns into a `patterns` module.

#### [CONV-004] Duplicate Type Names Across Crates

### Error Handling

#### [ERR-001] Production unwrap()/expect() Count: 19
- **Severity**: HIGH
- **Location**: ecc-domain (3), ecc-app (12), ecc-infra (2), ecc-cli (2)
- **Principle**: No `.unwrap()` or `.expect()` in production code (per `rules/rust/coding-style.md`)
- **Evidence**: 19 unwrap/expect calls in production code paths (excluding test modules). Most are `Regex::new().expect("valid regex")` which are safe (compile-time-known patterns) but violate the project rule.
- **Risk**: Any incorrect regex pattern causes a panic in production instead of an error.
- **Remediation**: For compile-time-known regex patterns, use `LazyLock<Regex>` (panic at startup, not at call time). For the remaining 3 in domain, audit and convert to `Result` propagation.

#### [ERR-002] Swallowed Errors in Drift Check
- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/src/drift_check.rs:57-59`
- **Evidence**: `let _ = fs.create_dir_all(parent)` and `let _ = fs.write(report_path, &report_content)` -- file system errors silently discarded.
- **Risk**: Drift check report may silently fail to write. User gets no feedback.
- **Remediation**: Propagate errors or log warnings.

#### [ERR-003] Swallowed Errors in Update Swap
- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/src/update/swap.rs:29,160,218-222`
- **Evidence**: Multiple `let _ = fs.write(...)` and `let _ = fs.remove_file(...)` calls. Test fixtures use `let _ =` appropriately, but line 29 (`let _ = restore_backup(...)`) in production code silently discards backup restoration failure.
- **Risk**: Failed rollback after botched update goes unnoticed.
- **Remediation**: Log or propagate backup restoration errors. At minimum, emit a warning.

#### [ERR-004] Backlog Lock Removal Swallowed
- **Severity**: MEDIUM
- **Location**: `crates/ecc-app/src/backlog.rs:215`
- **Evidence**: `let _ = locks.remove_lock(backlog_dir, &id)` -- lock cleanup failure ignored.
- **Risk**: Stale lock files may accumulate if removal fails.
- **Remediation**: Log a warning on lock removal failure.

#### [ERR-005] Error Type Granularity: Good
- **Severity**: INFO (positive)
- **Evidence**: Each module defines its own error enum using `thiserror`. Error types are well-structured with contextual information. 48 error types is appropriate for a codebase of this size and architecture.
